## Идея проекта

Программа рассчитывает, сколько поддонов и/или деталей можно разместить в грузовом автомобиле, и строит оптимальную схему раскладки (задача трёхмерной упаковки, 3D bin packing).

### Функциональные требования

1. **Транспортные средства.** Пользователь создаёт и сохраняет варианты грузовых автомобилей, задавая внутренние размеры грузового отсека: длину, ширину, высоту. Справочник типовых кузовов + свои варианты.
2. **Грузы (детали и поддоны).** Для каждого типа груза — длина, ширина, высота, количество единиц. В одной загрузке — грузы нескольких разных типов и размеров.
3. **Правила размещения.** Для каждого типа груза:
   - **вложение (nesting):** могут ли единицы вкладываться друг в друга, и какой эффективный прирост высоты даёт каждая вложенная единица;
   - **штабелирование (stacking):** можно ли ставить единицы друг на друга, ограничение по ярусам / по максимальной высоте штабеля;
   - **вращение (rotation):** только вокруг вертикальной оси (обмен L↔W), все 6 ориентаций, либо запрещено.
4. **Расчёт раскладки.** По машине и списку грузов строится оптимальная раскладка: максимизация количества размещённых единиц (или полноты заявки) с учётом всех правил.
5. **Результат.** Итоговое количество размещённого/неразмещённого по типам; схема раскладки (координаты + ориентация); показатель заполнения объёма; визуализация (минимум послойная 2D, в идеале 3D).

### Архитектурные и процессные требования

6. **Headless-движок.** Ядро — отдельная библиотека/пакет без привязки к UI, чистый JSON-API. Любой интерфейс (веб, CLI, MCP) — клиент этого API.
7. **Первый интерфейс — прототип в Lovable.** UI — обёртка над API движка, заменяемый слой. В спецификации зафиксировать контракт API между UI и движком.
8. **i18n с первого дня.** Никаких строк в коде: все тексты — через словари локалей. Стартовые языки — ru и en. Локализация единиц измерения и форматов чисел.
9. **Готовность к MCP (будущее).** API движка проектируется под публикацию как MCP-сервер (`list_vehicles`, `add_cargo`, `calculate_layout`, `get_layout_report`). Раздел «MCP-готовность» в спецификации. Реализация MCP в MVP не входит, контракты закладываются сейчас.
10. **Документация и реестр задач — постоянная обязанность.** В `docs/` актуальны spec, design, ADR (по файлу на решение), контракт API, changelog. Сначала документация, затем код. Реестр задач — только в beads; TODO в markdown и коде запрещены.

### Открытые вопросы — проработать в брейншторме

- Единицы измерения (мм/см/м), допуски и зазоры между грузами и стенками кузова.
- Учитывать ли вес: грузоподъёмность, максимальная нагрузка на нижний груз, распределение по осям.
- Алгоритм упаковки: эвристики (Extreme Points, Layer-based, guillotine, генетический) и компромисс скорость/качество.
- Тип приложения и стек: CLI, десктоп, веб; язык и фреймворк.
- Формат хранения справочников (JSON/БД), импорт грузов (CSV/Excel).
- Учёт порядка выгрузки (LIFO), «хрупкость» (нельзя ставить сверху), обязательная ориентация «этой стороной вверх».
- Поддоны как особый случай: груз на поддоне, стандартные размеры (EUR 1200×800, FIN 1200×1000).
- Критерий оптимальности: максимум единиц / максимум объёма / минимум машин.
- Требования к визуализации и экспорту (PDF, картинка, JSON).
- Граница Lovable/ядро: расчёт в браузере (npm/WASM) или на бэкенде (REST-API)?
- Объём i18n в MVP: только интерфейс или также отчёты/экспорт.
- Приоритет MCP: какие сценарии через агентов ожидаются.
