## Задача: `LKWkalk-4bj.6` — умная раскладка (последняя в эпике)

**БОЛЬШАЯ задача: нужны спека + ADR и решения владельца — код не писать, пока они не приняты.**

**Проблема (со скриншота прода):** shelf-эвристика режет полки под первый тип груза → в полках
остаются карманы (`×16`/`×4`/`×7`), ~84% пола, часть груза уходит в `unplaced`.

**Движок УЖЕ умеет, просто не выведено в UI** (это меняет оценку — половина задачи дешёвая):
- `Load.loadingMode: 'rear' | 'side' | 'combined'` (**ADR 012**, контракт 0.4.0, дефолт `combined`) —
  это и есть «ряды по ширине / по длине»: `rear` — полки во всю ширину, растут от передней стенки к
  задней двери; `side` — полосы во всю длину, растут от борта; `combined` — считает обе и берёт
  плотнейшую (детерминированный тай-брейк `rear`). Ёмкость одного типа режим не меняет — меняет
  раскладку и то, где оказывается остаток.
- Зоны по заказам — `CargoType.orderId` (**ADR 011**, контракт 0.3.0): позиции одного заказа кладутся
  смежной зоной (чтобы выгрузить заказ целиком), ориентация — «по макс-влезанию».

Т.е. часть работы = «включить существующее и дать выбор» (дёшево), часть = **новая эвристика
max-компакт против карманов** (дорого). Разведи это на подзадачи в beads.

**Вынеси владельцу на решение** (сразу с предложенным дефолтом, диалог не разводить): набор стратегий
и их названия в UI; дефолтная стратегия; как выбирается вариант — авто-лучший или галерея вариантов с
метриками; **допустимо ли ради плотности ломать смежность зон заказов** (ADR 011: заказ выгружается
целиком — это продуктовый компромисс, а не техническая деталь).

**Инварианты:** `findGeometryViolations(load, layout) === []` на каждом результате; правила
nesting/stacking/rotation не нарушаются ради заполнения; расчёт детерминирован при одинаковом вводе;
вид сбоку (конвенция `rowY`/depth) не рушится; ручные правки (`editLayout.ts`) продолжают работать.

**Открыто ещё (owner-side):** `s17` deep-link ERPNext, `k06` поля ERPNext, `zbi` бэкап тома, `i6b` Basic Auth.
Бэклог движка: `qrd.15` (экспорт PDF/PNG), `qrd.29` (дефолт nestingMode в ядре), `qrd.31` (clearance на позицию).

## Подводные камни, добытые кровью (не наступай)

- **Chrome НЕ печатает SVG `<pattern>`** — заливка/штриховка исчезают на бумаге. Штриховка заказов
  рисуется прямыми `<line>`/`<circle>` (`HatchMarks` в `lib/swatch.tsx`, аналитический клиппинг
  Лианга-Барски). `var()` резолвится только на direct-элементах. Не возвращать `<pattern>` в печать.
- **click-outside слушать `click`, НЕ `mousedown`** — иначе закрытие панели сдвигает DOM до `mouseup`,
  браузер шлёт click на предка, и кнопка («+ Position») не срабатывает.
- **Конвенция вида сбоку:** наблюдатель у нижнего края вида сверху (`y = width`) → **больший `y` = ближний
  ряд** (`depth 0`, полная непрозрачность), меньший `y` — дальше (тусклее). Поле `CutRect.rowY`.
- **`findGeometryViolations` пропускает пары одной колонки** (`cargoTypeId+x+y`) — для ручных правок
  нужна отдельная проверка футпринтов (`editLayout.ts`, `overlapsOtherStack`).
- **Ручной поворот — только yaw** (`lwh`↔`wlh`, ADR 013): опрокидывание меняет `dz` → z всех ярусов
  стопки становятся неверными (это перерасчёт, а не ручная правка). Якорь — угол `(x,y)`.
- **Длинные RU-лейблы ломают вёрстку** — контролы с текстом ограничивать по ширине + `truncate`;
  строка позиции `xl:flex-nowrap`, ниже — перенос.
- **`localStorage`-ключи:** `ladungsplaner.setup` (форма), `ladungsplaner.load` (план),
  `ladungsplaner.locale` (язык) — кэш черновика, НЕ источник истины (импорт ERPNext перезапишет,
  Reset очистит). `ladungsplaner.palletPresets` — каталог пользовательских пресетов, **переживает
  Reset**. `test-setup.ts` чистит localStorage в `afterEach`.
- **dist в .gitignore** — на чистой машине `npm ci` + сборка пакетов, иначе web/server-тесты не видят
  `@shadrin-v/engine`/`i18n`.
- **Docker-демон может быть выключен** — тогда локальный `docker build`-гейт не отработает; фактическую
  сборку образа проверит успешный деплой Coolify.
- **jsdom без layout** — баги, зависящие от вёрстки или реальной геометрии указателя, тестами не
  воспроизводятся; тест сторожит намерение, а не сам симптом.

принятия решений. Работаем по одной задаче: ветка → зелёные гейты → merge → деплой → проверка на проде.
