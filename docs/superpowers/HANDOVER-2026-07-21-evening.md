# HANDOVER 2026-07-21 (вечер) — каталог в браузере + расхождение прода

## Что сделано: `LKWkalk-7wb` (PR #11, `371041e`)

Каталог артикулов не работал в браузере вообще: ни одного запроса к `/api/`, сохранённые артикулы
никогда не предлагались, видны были только встроенные пресеты поддонов.

**Диагноз из беда оказался неверным.** `dp` из контекста был исправен всегда — проводка в
`apps/web/src/main.tsx` в порядке. Ломался сам запрос: `HttpDataProvider` по умолчанию клал в поле
голый глобальный `fetch` и звал его как собственный метод, то есть с провайдером в роли получателя.
Chrome на это отвечает `Illegal invocation` **до** обращения к сети, а общий `catch` в
`ArticleCombobox` ошибку гасил.

Почему 573 теста молчали: `fetch` в Node получателя не проверяет, а в jsdom-тестах провайдер
подставляется явно — мимо дефекта.

Проверено в настоящем Chrome через CDP, причинность в обе стороны: без фикса — ноль запросов и
`aria-expanded=false`; с фиксом — `GET /api/articles?q=Q` и подсказка из SQLite.

Вошло: вызов `fetch` с глобальным получателем (стрелка, а не `bind` — чтобы подмена
`globalThis.fetch` в тестах работала при провайдере, созданном на загрузке модуля); регрессионный
тест, проверяющий получателя **первого** вызова; логирование подавленной ошибки поиска в консоль.

574 теста, typecheck и lint чистые.

## Главное открытие сессии: прод отстаёт от `main` на 77 коммитов (`LKWkalk-cn9`, P1)

Жалоба «на проде не работает перемещение группой» — не баг кода. Coolify собирает приложение
с ветки **`production`** (`docs/INFRASTRUKTUR-ladungsplaner.md` §1.2), а вся работа мержится
в `main`. `production` стоит на `0f32ca6` и отстаёт на 77 коммитов.

Подтверждение: `GET /api/health` на проде отдаёт `contract 0.13.0`, в `main` —
`ENGINE_CONTRACT_VERSION = 0.14.0` (групповые операции появились именно в `0.14.0`).

Значит, на проде нет ничего из последних трёх сессий: группового переноса (`dwc.6`), удаления
позиции из расчёта (`yxn`), каталога артикулов (`7wb`).

**Выкладка безопасна:** единственное изменение схемы БД за 77 коммитов — `CREATE TABLE IF NOT
EXISTS article`, `migrate()` выполняется на старте, данные в томе `/app/data` сохраняются.
`ERPNEXT_*` можно по-прежнему не задавать — `/api/orders` ответит `503 ERR_ERPNEXT_UNCONFIGURED`,
остальное работает.

Системная причина, а не разовая оплошность: шаг «выложить на прод» нигде не описан как часть
процесса — ни в `CLAUDE.md`, ни в чеклисте завершения ветки. Поэтому расхождение копилось молча.
Заведено отдельно: `LKWkalk-7jg`.

## Грабли, к прошлым в дополнение

1. **Диагноз в беде — гипотеза, а не факт.** «`dp` в рантайме `null`» звучало убедительно и было
   неверным. Симптом (ноль запросов) одинаков и при отсутствии провайдера, и при падении `fetch`
   до сети. Различает их только проверка на живом коде.
2. **Node не проверяет получателя `fetch`, браузер проверяет.** Любой глобальный DOM/Web-API,
   сохранённый в поле объекта и вызванный как метод, в браузере даёт `Illegal invocation`.
   Оставшееся место того же класса — `apps/server/src/erpnext/adapter.ts:33` (`LKWkalk-8vy`,
   сейчас не дефект).
3. **Тест, гасящий ошибку, гасит и диагностику.** `catch { items = [] }` без единой строки в
   консоль превратил однострочный баг в многочасовое расследование. Подавляешь ошибку для
   пользователя — оставь её разработчику.
4. **CDP-скрипт может врать.** Мои первые прогоны показывали «ноль запросов» уже после фикса —
   печать шла в документ, который успевал перезагрузиться. Прежде чем верить отрицательному
   результату браузерной проверки, убедись, что страница та же самая.
