# Дизайн: кастом-поля габаритов в ERPNext (для импорта в Ladungsplaner)

Статус: утверждён пользователем (2026-07-14, брейншторм). Реализация адаптера — задача `LKWkalk-uvf`.
Связанные документы: [ADR 015](../../adr/015-fullstack-app-erpnext.md) ·
[дизайн-спека приложения](2026-07-14-fullstack-app-erpnext-design.md) ·
[api-contract движка](../../api-contract.md).

## 1. Контекст и решение

Приложению для расчёта раскладки нужны **габариты груза** (длина×ширина×высота). Товары ERPNext (домен
Schäfer) — это сами паллеты. Изначально планировался фолбэк-парсинг габаритов из названия товара
(«600x800 mm»); **это решение отменено** (2026-07-14): габариты приходят из **явных кастом-полей**
ERPNext, которые владелец создаёт на своей стороне. Парсер названий не пишем.

**Пока эти поля не заполнены — приложение не блокируется:** позиция импортируется с пустыми
габаритами, помечается `manual`, и пользователь вводит размеры в экране «Настройка».

## 2. Поля (3 шт.)

Тип — **Int**, значения в **целых миллиметрах** (внутренний формат движка, ADR 002; без конвертации).

| fieldname | label (de) | Тип | Обяз. | DocType |
|---|---|---|---|---|
| `custom_length_mm` | Länge (mm) | Int | нет | Item **и** Sales Order Item |
| `custom_width_mm`  | Breite (mm) | Int | нет | Item **и** Sales Order Item |
| `custom_height_mm` | Höhe (mm)  | Int | нет | Item **и** Sales Order Item |

- **Мастер — на `Item`** (габариты = свойство паллеты). Секция формы Item: группа
  «Ladungsplaner / Abmessungen».
- **Копия — на `Sales Order Item`**, каждое поле с **Fetch From**:
  - `custom_length_mm` → Fetch From `item_code.custom_length_mm`
  - `custom_width_mm`  → Fetch From `item_code.custom_width_mm`
  - `custom_height_mm` → Fetch From `item_code.custom_height_mm`
  - рекомендуется **Read Only** на строке (значение ведётся в Item).
- Поля **необязательные** — существующие товары пока без габаритов; заполняются постепенно.

## 3. Почему копия в строку (Fetch From), а не дочитывание Item

Читаем габариты **из строки Sales Order** — тогда адаптер берёт всё из **одного** запроса
`GET /api/resource/Sales Order/{name}` (в ответе уже есть `items[]` со всеми полями строки), без
запроса на каждый Item (без N+1). Значение в строке — **снимок на момент выбора товара в заказ**, что
согласуется со снимок-семантикой сохранённого плана (`Load`+`Layout`).

**Нюанс миграции:** `fetch_from` копирует значение **при выборе товара в строку**, не задним числом.
- Заказы, созданные **до** появления полей, в строке будут **пустыми** даже после заполнения Item →
  импорт пометит позиции `manual` (ручной ввод).
- Заказы, созданные **после** — заполняются автоматически.
- (Опционально, на стороне ERPNext: разовый бэкфилл строк старых заказов скриптом — вне объёма этой
  спеки.)

## 4. Контракт адаптера (`importOrder`, задача uvf)

Вход: `erpOrderId` (напр. `SAL-ORD-2026-00001`). Выход: `OrderZone` (см. `@shadrin-v/contracts`).

Маппинг `Sales Order` → `OrderZone`:
- `orderId` ← `data.name`.
- для каждой `data.items[i]` → `OrderPosition`:
  - `itemCode` ← `item.item_code`; `itemName` ← `item.item_name`; `quantity` ← `item.qty`;
  - если **все три** `custom_length_mm`/`custom_width_mm`/`custom_height_mm` присутствуют и > 0:
    `length/width/height` ← эти значения, `dimensionsSource = 'erpnext-field'`;
  - иначе: `length/width/height` не заданы (`undefined`), `dimensionsSource = 'manual'`.
- **Никакого парсинга названия.** Импорт не падает из-за отсутствия габаритов.

### Изменение контракта `@shadrin-v/contracts`

`DIMENSION_SOURCES` сокращается до **двух** значений:
```ts
export const DIMENSION_SOURCES = ['erpnext-field', 'manual'] as const;
```
(Убираются `parsed-name` — парсера нет — и `unknown` — схлопывается в `manual`.)

**Провенанс = только происхождение.** Признак «нужно заполнить» приложение выводит из **пустоты**
габаритов (`length/width/height === undefined`), а не из метки. Поэтому двух значений достаточно.

## 5. Поведение приложения (экраны, задачи gxp/73u)

- Позиции с `dimensionsSource='erpnext-field'` показываются с готовыми размерами.
- Позиции с пустыми габаритами (`manual`) — подсвечены как «требует ввода»; движок (`calculateLayout`)
  запускается только когда у всех позиций заданы размеры (геометрия без них невозможна).
- Правила груза (штабелирование/вложение/вращение, `state`) задаются и редактируются **в приложении**,
  дефолты — на стороне UI; в ERPNext их не выносим (см. §6).

## 6. Вне объёма (YAGNI сейчас)

- Правила груза как поля ERPNext (`stacking`/`nesting`/`rotation`) — **не выносим**; при необходимости —
  отдельная спека/итерация.
- Вес: `Item.weight_per_unit` в ERPNext уже есть, но вес — вне MVP движка (не читаем).
- Запись плана обратно в ERPNext (ссылка/вложение к Sales Order) — позже.
- Бэкфилл габаритов в старые заказы — на стороне ERPNext, вне этой спеки.

## 7. Чек-лист настройки на стороне ERPNext (для владельца)

1. `Item` → Customize Form (или Custom Field): добавить `custom_length_mm`, `custom_width_mm`,
   `custom_height_mm` (Int), секция «Ladungsplaner / Abmessungen».
2. `Sales Order Item` → добавить те же три поля (Int, Read Only), каждому задать **Fetch From** =
   `item_code.custom_<...>_mm`.
3. Проставить габариты у актуальных паллет-товаров (Item). Новые заказы подхватят значения
   автоматически.
