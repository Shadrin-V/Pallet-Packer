# Спек: Continuous deployment из `main` (закрепление шага релиза)

- **Bead:** `LKWkalk-7jg` (P2)
- **Дата:** 2026-07-22
- **Статус:** утверждён владельцем, готов к плану

## Проблема

Прод собирается Coolify с ветки `production`, а вся работа мержится в `main`. Шаг
«синхронизировать `production`» нигде не описан как часть процесса, поэтому расхождение
копилось молча: к 2026-07-21 `production` отставал от `main` на 77 коммитов (три сессии
работы, невидимой на проде — групповой перенос, удаление позиции, каталог артикулов).
Это системная причина (пропущенный шаг процесса), а не разовая оплошность.

## Решение

Перейти на **continuous deployment из `main`**: Coolify собирает ветку `main`, ветка
`production` ретайрится. Расхождение исключено by design — ветки, которую можно забыть
синхронизировать, больше нет. Инвариант: **`main` всегда прод-готов; «шаг релиза» = мерж в
`main` (при зелёном CI) + проверка live**.

Развилки, зафиксированные владельцем в брейншторме:
- **Модель релиза:** автоматически после каждого мержа (не явный релиз по команде, не
  полу-авто через PR). Прод следует за `main`.
- **Docs-only мержи:** тоже вызывают пересборку — принято ради простоты (ноль новой
  автоматики, никаких path-фильтров и GitHub Actions для деплоя).
- **CI-гейт:** входит в эту задачу (не отдельным bead) — CD безопасен ровно настолько,
  насколько зелёный `main`.

## Компоненты

### 1. Coolify: сменить ветку сборки (owner-side, панель Coolify)
В приложении Ladungsplaner сменить **Branch: `production` → `main`**. Webhook GitHub шлёт
все пуши, Coolify фильтрует по своей настройке ветки — смены настройки достаточно, webhook
переконфигурировать не нужно. После переключения проверить тривиальным пушем в `main`, что
билд стартует и `/api/health` остаётся здоровым.

### 2. Ретайр ветки `production`
Строго **после** переключения Coolify (иначе первый пуш в `main` не задеплоит, а webhook,
привязанный к удалённой ветке, сломается):
```bash
git push origin --delete production
git branch -D production   # локально
```

### 3. Документация (репо)
- **Новый `docs/adr/023-continuous-deploy-from-main.md`** — по шаблону контекст → решение →
  последствия:
  - *Контекст:* расхождение 77 коммитов, пропущенный шаг релиза.
  - *Решение:* Coolify собирает `main`; `production` ретайрится; `main` всегда прод-готов;
    шаг релиза = мерж в `main` + проверка live; гейт качества — CI перед мержем.
  - *Последствия:* нечего забывать синхронизировать; docs-мержи тоже пересобирают прод
    (принято); откат — через панель Coolify (redeploy предыдущего деплоя), отдельной
    ветки-буфера нет; безопасность CD держится на зелёном CI-гейте (компонент 4).
- **`docs/INFRASTRUKTUR-ladungsplaner.md`** — §0 (строка «Ветка автодеплоя») и §1.2
  («Branch: production») → `main`; добавить строку «CD: мерж/пуш в `main` = live,
  отдельного шага релиза нет; откат — redeploy в панели Coolify».
- **`docs/adr/015-fullstack-app-erpnext.md`** строка про ветку `production` → пометка
  «в части ветки деплоя заменено ADR 023».
- **`docs/superpowers/START-PROMPT.md`** — убрать инструкции «push `main` + `production`»
  (строки ~8, 25, 95); заменить на «мерж в `main` → Coolify авто-деплоит».
- **`.claude/commands/wrap.md`** и **`.claude/commands/start.md`** — если упоминают
  push/pull `production`, убрать.
- **`CLAUDE.md`** — в разделе «Session Completion» и в строке *Release* таблицы «Этап 2»:
  прод деплоится автоматически при мерже в `main`; «шаг релиза» = мерж в `main` (CI зелёный)
  + проверка live; отдельной ветки `production` нет.
- **`docs/CHANGELOG.md`** — запись о переходе на CD из `main`.

Исторические `docs/superpowers/HANDOVER-*.md` — дневниковые снимки, **не трогаем**.

### 4. CI-гейт: `.github/workflows/ci.yml`
- **Триггеры:** `pull_request` с таргетом `main`; `push` в `main`.
- **Concurrency:** группа по `${{ github.ref }}`, `cancel-in-progress: true`.
- **Job `ci`** (ubuntu-latest, Node **22** — совпадает с Dockerfile):
  1. `actions/checkout`
  2. `actions/setup-node@v4` (node-version 22, `cache: npm`)
  3. `npm ci` (корень; workspaces-монорепо ставит все пакеты)
  4. `npm run typecheck`
  5. `npm run lint`
  6. `npm test` (`vitest run`, 574+ тестов)
  7. `npm run build` (ближайший прокси к «деплоемый артефакт собирается»; ловит ошибки
     vite/tsc до мержа, а не после — в отличие от билда в Dockerfile, который падает уже
     после ухода на прод)

### 5. Branch protection на `main`
Требовать прохождение статус-чека `ci` перед мержем — иначе CI *информирует*, но не
*блокирует*. Настроить через `gh api` (репо `Shadrin-V/Pallet-Packer`), либо владелец в
Settings → Branches. Правило: required status check `ci`, применяется к `main`.

## Тестирование / приёмка

Задача процессно-инфраструктурная, поэтому «тест» — это проверяемые артефакты и live-факты,
а не юнит-тесты:
- **CI:** после открытия PR с этими изменениями workflow `ci` запускается и проходит
  зелёным (виден в GitHub Actions). Проверить, что он также триггерится на `pull_request`.
- **Branch protection:** попытка мержа при красном `ci` блокируется (или: в настройках
  ветки виден required check `ci`).
- **Coolify:** после переключения ветки — тривиальный пуш в `main` даёт новую сборку;
  `GET https://ladungsplaner.holz-schaefer.de/api/health` отвечает и `contract` не
  откатывается.
- **Ретайр:** `git ls-remote --heads origin production` пуст; `git branch -a` без
  `production`.
- **Документация:** grep по активным докам (`docs/INFRASTRUKTUR-*`, `START-PROMPT.md`,
  `CLAUDE.md`, `.claude/commands/`) не находит инструкций деплоить `production`.

## Порядок и риски

Порядок критичен: **сначала CI + branch protection + документация (обычный PR в `main`)**,
и только **после** мержа этого PR — owner-side переключение Coolify на `main`, затем ретайр
`production`. Если переключить Coolify раньше, чем в `main` появится CI, короткое окно
«CD без гейта» всё же будет, но это разовое окно на одну сессию, а не постоянное состояние.

Риск отката: без ветки-буфера `production` откат делается через панель Coolify (redeploy
предыдущего успешного деплоя). Это задокументировано в ADR-023 и рунбуке.

## Вне объёма

- Path-фильтры / пропуск docs-мержей при деплое (владелец выбрал «пересобирать всегда»).
- Полу-авто PR `main → production` и явный релиз по команде (отвергнуты в пользу CD).
- Staging-окружение и blue-green деплой (YAGNI для внутреннего инструмента).
