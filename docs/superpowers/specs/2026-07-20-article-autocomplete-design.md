# Дизайн: единое поле-автокомплит по артикулам в строке позиции

Статус: DESIGN — утверждён владельцем 2026-07-20. bd: `LKWkalk-8a0` (закрывает заодно `LKWkalk-rgv.8`).

## Проблема

В строке позиции экрана «Настройка» сегодня три пересекающихся элемента: `Select` с пресетами
и пунктом «Свой размер», отдельное поле названия и три поля габаритов. Название дублируется
(пресет + имя идут двумя полями подряд — `rgv.8`), а правила груза — вложение, шаг вложения,
ярусы, ориентация, ось вил — вводятся заново при каждой заявке, хотя для конкретного артикула
они неизменны. Кнопка «Сохранить пресет» существует, но спрятана в панели деталей, хранит
только габариты, живёт в `localStorage` и не знает про артикулы.

## Решение

Одно поле-комбобокс «Артикул» вместо пресета и названия. Ввод даёт подсказки из каталога
артикулов; выбор подставляет габариты и правила целиком. Каталог хранится в SQLite на
`apps/server`, пополняется кнопкой «Сохранить артикул в базу» и автоматически при импорте
заказов из ERPNext.

## Принятые решения

| № | Вопрос | Решение |
|---|--------|---------|
| 1 | Источник подсказок | Локальный каталог на сервере. ERPNext — источник истины для конструктивных полей, синхронизация по кнопке появится позже |
| 2 | Обязателен ли артикул | В строке ввода — нет (разовый груз считается без сохранения). Для сохранения в базу — да: без артикула запись не сматчится при синке |
| 3 | Когда делаем синк с ERP | Отдельной задачей после `LKWkalk-k06` (кастом-поля габаритов в ERPNext ещё не созданы). Схема закладывается сразу под ERP-приоритет |
| 4 | Что подставляет подсказка | Название, габариты и все правила. Количество — нет, оно из заявки |
| 5 | Запирание конструктивных полей | Заперты, только когда значение пришло из ERP. Пустое поле вводится руками без ошибки; следующий синк перезапишет и запрёт |
| 6 | Сколько конструктивных приростов | Два: толщина верхнего настила (парное вложение) и прирост «один в один». Оба конструктивные |

## Доменная поправка (вопрос 6)

Прирост высоты при вложении — конструктивное свойство паллеты, как и габариты, и зависит от
режима вложения:

- **парное вложение** — прирост равен толщине верхнего настила;
- **«один в один»** — прирост задан конструкцией и отличается от первого.

В ядре сегодня одно поле `nesting.stepHeight` (`packages/engine/src/model/types.ts:36`), смысл
которого плавает по режиму, — переключение режима молча оставляет цифру от другого режима и
высота стопки считается неверно. **Ядро и контракт API не меняются:** два прироста хранит
каталог, а форма подставляет в `stepHeight` тот, который соответствует выбранному режиму.
ADR не требуется.

## Модель данных

Новая таблица в `apps/server/src/db/schema.ts`, рядом с `vehicle` и `loading_plan`:

```
article
  item_code             TEXT PRIMARY KEY   -- артикул, ключ синхронизации с ERP
  name                  TEXT NOT NULL
  length, width, height INTEGER NULL       -- конструктивные, целые мм
  nest_step_pairwise    INTEGER NULL       -- толщина верхнего настила
  nest_step_sequential  INTEGER NULL       -- прирост «один в один»
  rules_json            TEXT NOT NULL      -- свободные параметры
  source                TEXT NOT NULL      -- 'erp' | 'local'
  synced_at             TEXT NULL
  updated_at            TEXT NOT NULL
```

**Конструктивные поля** — `length`, `width`, `height`, `nest_step_pairwise`,
`nest_step_sequential`. Заперты от правки, когда `source='erp'` и значение не `NULL`.
`NULL` означает «в ERP ещё не заполнено» — поле правится руками, ошибки нет.

**Свободные поля** (`rules_json`) — `state`, `nestingMode`, `maxNested`, `allowUnpairedTop`,
`maxTiers`, `rotation`, `forkAccess`, `forkAxis`. Всегда локальные; ERP их не знает и никогда
не перезапишет.

`source` и `synced_at` заводятся сразу, хотя кнопки синка в этой задаче нет, — чтобы задача
после `k06` свелась к одному роуту, а не к миграции схемы.

## API

В `DataProvider` (`apps/web/src/data/DataProvider.ts`) добавляются два метода — SPA
по-прежнему не ходит в ERPNext напрямую:

```ts
searchArticles(query: string): Promise<Article[]>;   // GET /api/articles?q=
upsertArticle(a: ArticleInput): Promise<Article>;    // PUT /api/articles/:itemCode
```

Поиск выполняется на сервере (`LIKE`, лимит 20 записей), порядок выдачи:

1. точное совпадение артикула;
2. артикул начинается с введённого;
3. введённое встречается в названии.

Регистр не учитывается. Нечёткий поиск с исправлением опечаток не делаем: артикулы копируют
из заявки, а не печатают по памяти.

Будущий `syncArticlesFromErp()` добавится третьим методом, не меняя эти два.

## Автопополнение из заказов

При `importOrder` сервер делает upsert артикулов заявки: `item_code`, `item_name` и габариты,
если кастом-поля ERPNext заполнены, с `source='erp'`. Свободные правила не затрагиваются:
однажды заданное для артикула вложение переживает импорт следующего заказа.

## UI строки позиции

`Select` пресетов и `TextField` названия заменяются одним комбобоксом «Артикул»
(`apps/web/src/screens/SetupScreen.tsx`). Это закрывает `rgv.8`.

- Подсказка в списке: артикул полужирным, название рядом, габариты справа мелким, пометка
  источника «ERP» или «локально».
- Ввод свободный: ничего не выбрано — введённый текст становится названием позиции, габариты
  вводятся руками, расчёт работает.
- Выбор подсказки подставляет название, габариты, оба прироста и все правила.
- Конструктивные поля из ERP — read-only с пометкой и подсказкой «меняется в ERPNext,
  артикул такой-то». Пустые из ERP — обычные поля ввода.
- Переключение режима вложения подставляет в `stepHeight` ядра прирост своего режима.
- Клавиатура: `↑`/`↓` по списку, `Enter` — выбрать, `Esc` — закрыть. `role="combobox"` +
  `aria-activedescendant`; существующие тесты по `ariaLabel` продолжают работать.

**Кнопка «Сохранить артикул в базу»** заменяет «Сохранить пресет» в панели деталей — в самой
строке места нет, она упирается в ширину карточки. Активна при введённом артикуле и
заполненных габаритах. Если артикул уже в базе, надпись — «Обновить артикул»; перезаписываются
свободные правила и только незапертые конструктивные поля.

Все строки — через ключи локалей (`de`, `ru`), как требует `CLAUDE.md`.

## Что удаляется

`apps/web/src/data/userPresets.ts` (каталог в `localStorage`) удаляется целиком: его записи не
имеют артикула и не могут участвовать ни в подсказках, ни в синхронизации. Сохранённые
пользователем пресеты при этом пропадают — автоматическая миграция невозможна, артикул им
взяться неоткуда; владелец переносит единичные записи руками.

Встроенные `PALLET_PRESETS` (EPAL 1/2/3/6, четвертная) остаются статическим источником
подсказок с пометкой «стандарт»: артикула у них нет, в базу они не сохраняются, но выбрать
EPAL 1 можно в два символа.

## Тестирование

**Ядро** — не меняется, тесты не трогаем.

**Сервер:**
- порядок выдачи поиска (точное → префикс → название), лимит 20, регистронезависимость;
- `upsert` создаёт новую запись и обновляет существующую;
- импорт заказа пополняет каталог и не затирает локальные правила артикула.

**Web:**
- ввод даёт релевантные подсказки, выбор подставляет габариты и правила;
- поле с габаритом из ERP не редактируется; пустое поле из ERP редактируется;
- переключение режима вложения подставляет прирост своего режима;
- «Сохранить артикул в базу» создаёт запись, видимую в подсказках после перезагрузки;
- клавиатурная навигация по списку подсказок.

## Вне рамок

- Кнопка синхронизации с ERPNext — отдельная задача, зависит от `LKWkalk-k06`.
- Нечёткий поиск и исправление опечаток.
- Перенос пользовательских пресетов из `localStorage`.
