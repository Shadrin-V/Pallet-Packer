# Дизайн: qrd.4 — ShelfPacker (floor-стадия)

Дата: 2026-07-10 · Задача: `LKWkalk-qrd.4` · Статус: одобрено пользователем (brainstorming)

Переписывание WIP-упаковщика (`wip/qrd-4`, коммит `49f72cb` «old orientation heuristic — needs
rewrite») под [ADR 011](../../adr/011-order-grouping.md) (макс-влезание) и [ADR 012](../../adr/012-loading-modes.md)
(режимы загрузки). Заменяет старую эвристику EUR=33.

## 1. Цель и граница

`packFloor` — **чистый геометрический примитив**: раскладывает набор футпринтов по прямоугольной
области пола, детерминированно, без понятия «заказ». Возвращает координаты и напольную ориентацию
каждой единицы.

**Граница qrd.4 / qrd.7 (решено пользователем):** зонирование по `orderId`, распределение заявки
(`fill`/`quantity`), сборка `Layout` с `contractVersion` и добавление полей `CargoType.orderId` /
`Load.loadingMode` в модель — всё это **оркестратор `LKWkalk-qrd.7`**. Оркестратор режет пол на зоны,
вызывает `packFloor` на каждую зону и сдвигает координаты на смещение зоны. Поэтому `packFloor`
работает от origin области `(0,0)` и ничего не знает про заказы.

Система координат: `x` — длина (`0` = передняя стенка, `L` = задняя дверь), `y` — ширина
(`0` = борт загрузки), `z` — высота (floor-стадия всегда `z = 0`, `tier = 1`).

## 2. Интерфейс

```ts
type LoadingMode = 'rear' | 'side' | 'combined';   // default 'combined'

interface FloorRequest {
  cargoTypeId: string;
  length: number;   // футпринт по «родной» длине, мм > 0
  width: number;    // футпринт по «родной» ширине, мм > 0
  rotation: RotationRule;   // 'none' | 'yawOnly' | 'full'
  count: number;    // сколько пытаться разместить (fill → большое число)
}

interface FloorPlacement {
  cargoTypeId: string;
  x: number; y: number;     // угол единицы, мм, от origin области
  dx: number; dy: number;   // занятые размеры по осям x/y после выбора ориентации
  orientation: 'lwh' | 'wlh';
}

function packFloor(
  region: { length: number; width: number },
  requests: FloorRequest[],
  opts?: { clearance?: number; loadingMode?: LoadingMode },
): FloorPlacement[];
```

`unplaced` **не** возвращается явно: футпринты одного типа взаимозаменяемы, оркестратор выводит
неразмещённое, сравнив `count` с числом размещённых на `cargoTypeId`.

## 3. Алгоритм

### 3.1 Ориентация — «по макс-влезанию», на уровне типа
Для типа с футпринтом `l×w` рассматриваются только две **напольные** (yaw) ориентации; высота `H`
единицы неизменна (переворот на грань — вне MVP, [ADR 012](../../adr/012-loading-modes.md)):

- `lwh`: `dx = l`, `dy = w`;
- `wlh`: `dx = w`, `dy = l` (поворот на 90° вокруг вертикали).

`wlh` разрешён при `rotation ∈ {yawOnly, full}`; при `none` — только `lwh`. Из разрешённых берётся
ориентация с бóльшим `N(o) = floor(L/dx) · floor(W/dy)` (гомогенная ёмкость области). Тай-брейк при
равенстве — `lwh`. Выбор ориентации **не зависит** от оси полки (см. 3.2), поэтому эталонные числа
одинаковы во всех режимах.

Проверка: область `13600×2430`.
- EUR `1200×800`: `lwh` → `11·3 = 33`; `wlh` → `17·2 = 34` ⇒ **wlh, 34**.
- Gitterbox `1240×835`: A → `10·2 = 20`; B → `16·1 = 16` ⇒ **20**.

### 3.2 Полки — shelf next-fit; ось зависит от режима загрузки
Одна активная полка. Внутри полки — next-fit по порядку входа (порядок `requests`, затем `count`
внутри запроса) — это приоритет: переполнение области → хвост списка остаётся неразмещённым.

- **`rear`:** полка — полоса во всю **ширину** `W`; единицы кладутся вдоль `y` (`0→W`); при
  переполнении по `y` полка закрывается (её глубина по `x` = max `dx` вошедших), следующая полка
  открывается дальше по `x` (`0→L`, от передней стенки к двери). Условие открытия полки:
  `x_cursor + depth ≤ L`.
- **`side`:** полка — полоса во всю **длину** `L`; единицы вдоль `x` (`0→L`); при переполнении по `x`
  полка закрывается (глубина по `y` = max `dy`), следующая открывается дальше по `y` (`0→W`, от
  борта). Условие: `y_cursor + depth ≤ W`.
- **`combined` (default):** прогнать `rear` и `side`, вернуть раскладку с бóльшим числом размещённых
  единиц; при равенстве — `rear` (детерминированный тай-брейк).

Единица помещается в активную полку, если влезает по оси укладки **и** по глубине текущей полки не
выходит за область; иначе пытается открыть новую полку; если и это невозможно — не размещается.

### 3.3 Clearance
Равномерный зазор `clearance` (default 0). Вдоль любой оси `N` единиц требуют
`N·dim + (N−1)·clearance ≤ span`. Зазор применяется и между полками (глубина полки + clearance).

## 4. Детерминизм и краевые случаи

- Никакого `Math.random`; все размеры — целые миллиметры; порядок обхода фиксирован ⇒ одинаковый
  вход даёт **побайтово** одинаковый выход (в т.ч. порядок массива `FloorPlacement[]`).
- `count ≤ 0` — запрос пропускается.
- Футпринт больше области в обеих разрешённых ориентациях — 0 размещений для типа.
- `clearance`, съедающий место — меньше единиц или 0.
- Пустой `requests` → `[]`.

## 5. Тестирование (TDD)

Эталоны фиксируются как pinned-числа; раскладка проверяется валидатором геометрии
`findGeometryViolations`, а не сравнением картинок.

1. Тривиальный точный: область `2×2`, футпринт `1×1`, `clearance 0` → **4** размещения.
2. EUR `1200×800` `yawOnly` на `13600×2430` → **34** (ориентация `wlh`).
3. Gitterbox `1240×835` `yawOnly` → **20**.
4. `rotation:none` EUR → **33** (свап запрещён, `lwh`).
5. `rear`: гомогенный EUR = 34; координаты растут вдоль `x`; неразмещённый хвост у `x → L`.
6. `side`: EUR = 34; координаты растут вдоль `y`.
7. `combined`: на смешанном кейсе `placed(combined) ≥ max(placed(rear), placed(side))`; при равенстве
   раскладка совпадает с `rear`; детерминизм (два прогона идентичны).
8. Mixed (2 типа): нет geometry-violations; приоритет — первый тип размещается раньше второго при
   нехватке места.
9. `clearance` уменьшает число (< 34); `count` соблюдается при `count < ёмкости`.
10. **Property-based:** случайные футпринты/области/режимы → `findGeometryViolations(...) == []`
    (fast-check, если доступен в стеке; иначе seeded-цикл фиксированных случаев без `Math.random`).
11. Пустой `requests` → `[]`; футпринт больше области → 0 для типа.

## 6. Вне объёма qrd.4 (доковые артефакты и qrd.7)

- **Документация (сначала, до кода):** [ADR 012](../../adr/012-loading-modes.md) — режимы загрузки;
  контракт **0.4.0** — добавлен `Load.loadingMode`.
- **qrd.7 (оркестратор):** зонирование по `orderId`, распределение `fill`/`quantity`, сборка
  `Layout`/`Report`, бамп `ENGINE_CONTRACT_VERSION` в коде, поля `CargoType.orderId` /
  `Load.loadingMode` в модель.
- **Отложено:** гибрид смешанных стратегий в одной раскладке (разные зоны — разной осью); FFDH и
  skyline для плотности; переворот на грань (полные 6 ориентаций); учёт стороны двери/LIFO маршрута.
