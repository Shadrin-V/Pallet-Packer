# Бриф: дизайн-система Ladungsplaner (для отдельной сессии)

Статус: KICKOFF — задание для агента в **параллельной сессии**. bd: `LKWkalk-563`.
Дата: 2026-07-14. Владелец даст примеры сайтов и скриншоты дизайнов, которые ему нравятся.

> **Этот документ — задание, а не готовая дизайн-система.** Результат работы (готовые
> дизайн-документы) владелец приложит позже; они станут источником истины для экранов
> `LKWkalk-gxp` (Настройка) и `LKWkalk-73u` (Ladeplan), которые **заблокированы** до их появления.

## 1. Цель

Проработать **полную дизайн-систему** приложения Ladungsplaner (не косметика, а система):
токены, типографика, сетка/спейсинг, компоненты и их состояния, паттерны взаимодействия, печать A4.
Достаточную, чтобы по ней собрать два экрана-эталона на React без дизайнерских «додумок» в коде.

## 2. Что нравится / что НЕ нравится (важно, со слов владельца)

- ❌ **Реализация в Lovable — НЕ нравится и отвергнута** (ADR 015). Это внешний прототип-приложение;
  его внешний вид/вёрстку **не воспроизводить и не брать за ориентир**.
- ✅ **Нравятся мокапы, сделанные Claude в этом проекте** — это НЕ Lovable, это референс-макеты,
  собранные ранее в рамках проекта:
  - [`docs/lovable/setup-reference.html`](../../lovable/setup-reference.html) — экран «Настройка»;
  - [`docs/lovable/ladeplan-reference.html`](../../lovable/ladeplan-reference.html) — экран «Ladeplan»;
  - [`docs/lovable/design-system.md`](../../lovable/design-system.md) — извлечённые из них токены/рецепты.
  > Примечание: папка называется `docs/lovable/` по историческим причинам, но эти три файла —
  > **авторские макеты Claude, а не вывод Lovable.** Их и берём за отправную точку/направление.
- ➕ Владелец **дополнительно приложит** примеры сайтов и скриншоты понравившихся дизайнов —
  их нужно учесть и синтезировать с текущим направлением.

## 3. Процесс (обязательно)

1. Агент **ведёт брейншторм** (навык `superpowers:brainstorming`): вопросы **по одному**, и
   **задаёт их до тех пор, пока не сложится полная и однозначная картина** того, что хочет владелец
   (настроение/бренд, плотность, акценты, тёмная тема да/нет, типографика, поведение компонентов,
   мобайл да/нет, печать). Не переходить к спеке, пока картина неполная.
2. По итогам — **написать дизайн-спеку** (в `docs/superpowers/specs/`), утвердить с владельцем по
   разделам.
3. Обновить/заменить `docs/lovable/design-system.md` (или вынести в `docs/design/`) как канонический
   источник токенов и компонентов.
4. Терминал процесса — `writing-plans` для экранов не нужен здесь: реализацию экранов делает основная
   сессия (задачи gxp/73u) уже по готовой дизайн-системе.

## 4. Технические рамки (не менять без согласования)

- Стек экранов: **React (apps/web)**, **только токены** (ни одного hex в JSX), Tailwind-мост уже
  настроен на CSS-переменные (`apps/web/tailwind.config.js` + `theme.css`).
- **i18n**: немецкий по умолчанию + русский (`@shadrin-v/i18n`); ни одной строки в коде — только ключи.
- Движок `@shadrin-v/engine` переиспользуется (разрезы рисуются из `Layout`/`orientedDims`, штабель —
  `computeStack`). Дизайн не диктует доменную логику.
- Текущие допущения (можно пересмотреть в брейншторме): светлая тема, шрифт Inter, desktop-first,
  печать A4 для Ladeplan. Категориальная палитра заказов — цвет+штрих (обязателен штрих для ЧБ/дальтонизма).

## 5. Результат (Definition of Done)

- Дизайн-спека + канонический файл токенов/компонентов, утверждённые владельцем.
- Достаточно детально, чтобы `gxp`/`73u` собрались без дизайн-решений «на ходу».
- Владелец сообщает основной сессии, что документы готовы → снимаем блок с `gxp`/`73u`.
