# Групповое выделение и перенос стопок (LKWkalk-dwc.6)

> Дизайн утверждён владельцем 2026-07-21. Завершает эпик `LKWkalk-dwc` (буфер стопок).

## Задача

Стопки в плане двигаются только по одной. Нужно выделять несколько стопок рамкой мыши по виду
сверху и переносить их как единый блок — внутри кузова и одним движением в буфер/склад.

## Решения владельца

| Вопрос | Решение |
|---|---|
| Возврат группы из буфера | **Асимметрично**: в буфер группой, обратно поштучно. Буфер раскладывает тайлы построчно и формы группы не хранит; возвращать «точно как было» нечего. |
| Начало рамки | Протяжка с **пустого места** пола. Нажатие на стопку по-прежнему тащит стопку. |
| Правило захвата | **Касание**: пересечение рамки со стопкой достаточно. Отсек 13,6 м рисуется сжатым по ширине, требование полного охвата было бы мучительным. |
| Конфликт при броске | **Групповой магнит**: ядро ищет ближайшую дельту, при которой законна вся группа. Не нашлось в допуске — отказ целиком. |
| Визуализация | Обводка каждой выделенной стопки **и** охватывающая рамка группы со счётчиком. |

## Ядро: групповая алгебра правок

Аддитивное расширение контракта `0.13.0` → `0.14.0`, ADR 021. Документация пишется до кода.

```ts
unplaceStacks(load, layout, refs: StackRef[]): EditResult
moveStacks(load, layout, refs: StackRef[], dx: number, dy: number): EditResult
resolveGroupDrop(load, layout, refs, aim: { dx: number; dy: number }, opts?): GroupDropResolution

interface GroupDropResolution { dx: number; dy: number; ok: boolean; error?: EngineError; blocking: StackRef[] }
```

`moveStacks` принимает **дельту**, а не новые координаты. Это и есть жёсткий перенос: взаимное
расположение сохраняется по построению, а не проверкой.

### Групповой магнит

Обобщение существующего `resolveDrop` (ADR 020), не второй алгоритм. Одиночный магнит перебирает по
каждой оси кандидатов «прицел / обе стены / вплотную к каждому соседу», отбрасывает то, что дальше
допуска, и выбирает лучшего по правилу *вплотную важнее, чем близко*
(`packages/engine/src/packing/resolveDrop.ts:126-187`).

Групповой делает то же, но кандидат — **общая дельта**: для каждой выделенной стопки берутся её
собственные кандидаты, из них вычитается её прицельная позиция. Дельта годится, если при ней все
выделенные стопки в габаритах и не пересекаются с невыделенными. Между собой выделенные пересечься
не могут: они не пересекались до сдвига, а сдвиг общий.

Порядок вычислений задан производительностью — магнит считается на каждый `pointermove`. Кандидаты
сначала **оцениваются** (O(1) на кандидата после предрасчёта по осям), сортируются по «вплотную,
затем ближе», и только потом по порядку **проверяются** на валидность до первого прошедшего. Дорогая
проверка выполняется обычно один-два раза за движение мыши.

Отвергнутая альтернатива: вызывать `resolveDrop` по стопкам по очереди. Дёшево, но каждая стопка
притягивалась бы независимо и форма группы разъезжается — ровно то, что требование запрещает.

Допуск магнита (`opts.tolerance`) по умолчанию — минимум по выделенным стопкам от «половины
короткой стороны отпечатка», то есть то же правило, что у одиночного магнита, взятое по самой
тесной стопке группы: группа не должна притягиваться дальше, чем притянулась бы её самая
чувствительная участница. `blocking` — невыделенные стопки, мешающие в прицельной дельте.

### Отказ

Не нашлось дельты в допуске → раскладка не меняется, возвращается `ERR_EDIT_OVERLAP` или
`ERR_EDIT_OUT_OF_BOUNDS` и список мешающих стопок для подсветки. Как и вся остальная алгебра правок
(ADR 019): исходная раскладка плюс код ошибки, никогда полуприменённое состояние.

`unplaceStacks` геометрию не проверяет: снять стопки с пола можно всегда, поэтому «в буфер группой»
отказать не может в принципе. Инвариант сохранения `placed + unplaced` держится тем, что это свёртка
существующего `unplaceStack`.

## UI

### Состояние

В `CrossSection.tsx` `sel: StackSel | null` становится `sel: StackRef[]`, сравнение `sameStack` —
проверкой вхождения. Одиночное выделение перестаёт быть отдельным случаем: это массив из одного
элемента, поэтому обводка и ручка поворота работают без ветвлений «один или много».

### Разбор жеста на `pointerdown`

| Нажатие | Действие |
|---|---|
| По пустому полу, протащил дальше 40 мм | рисуется рамка; на отпускании — новое выделение по касанию |
| По пустому полу, без протяжки | выделение снимается |
| По стопке вне выделения | она становится единственным выделением и тащится (как сейчас) |
| По стопке внутри выделения | тащится вся группа |
| Shift/Ctrl-клик по стопке | добавить/снять её из выделения, не трогая остальных |

Порог 40 мм — существующий `CLICK_SLOP_MM`, тот же, что отделяет клик от перетаскивания. Один порог
на оба жеста, а не второе число рядом.

Escape снимает выделение, а во время протяжки отменяет рамку.

### Чистый модуль попадания в рамку

jsdom не реализует `getScreenCTM`, поэтому любой пиксельный жест в тестах схлопывается в ноль
(задокументировано в `CrossSection.test.tsx:198-200`). Попадание стопок в рамку обязано жить отдельно
от указателя: новый `apps/web/src/screens/components/marquee.ts` с `normalizeRect` и
`stacksInRect(rects: CutRect[], rect): StackRef[]` — по образцу `cutaway.ts` и `warehouseLayout.ts`.
Он покрывается тестами полностью; компоненту остаётся перевести пиксели в мм и позвать.

### Хром

Каждая выделенная стопка — своя пунктирная обводка `var(--brand)`, как сейчас. Поверх — тонкая
охватывающая рамка группы с подписью-счётчиком (`ladeplan.selection.count`, новый ключ локали de/ru).
Рамка выделения во время протяжки — полупрозрачный прямоугольник, `data-testid="marquee"`. Весь хром,
как и нынешний, помечается `className="print:hidden"`: на печать и в PDF не идёт.

Ручка поворота показывается только когда выделена ровно одна стопка. Поворот группы — отдельное
решение (крутить каждую стопку на месте или блок целиком?), в критериях приёмки бедa его нет, в эту
задачу не входит.

### Группа в буфер

Отпускание за пределами SVG над полосой склада уже хит-тестится через `bufferRef`
(`LadeplanScreen.tsx:245-251`); меняется только вызываемая функция — `unplaceStacks` вместо
`unplaceStack`. Выделение после этого очищается: стопок больше нет на полу. После успешного
группового переноса внутри кузова выделение, наоборот, сохраняется — чтобы можно было подвинуть ещё
раз, не выделяя заново.

Ошибки идут по существующему каналу `editError` → `EDIT_ERROR_KEYS` → `data-testid="edit-error"`.
Новых кодов не появляется.

## Тестирование

### Ядро

- `moveStacks` сохраняет взаимное расположение — попарные разности координат до и после идентичны;
- отказ целиком: группа из трёх стопок, одна из которых упёрлась бы в стену — раскладка **побайтово
  исходная**, `error.code = ERR_EDIT_OUT_OF_BOUNDS`. Проверяется именно неизменность раскладки, а не
  только наличие ошибки;
- `unplaceStacks` сохраняет баланс `placed + unplaced` по каждому типу;
- групповой магнит: группа, брошенная с зазором 60 мм от стены, встаёт вплотную;
- групповой магнит детерминирован: один ввод — одна дельта;
- вырожденные входы: пустой `refs`, `refs` с несуществующей стопкой, дельта `(0, 0)`;
- property-based (`fast-check`, уже в корневых devDependencies): после любого принятого группового
  сдвига `findGeometryViolations(load, result) === []`.

### Web

`marquee.test.ts` — касание углом попадает, рамка нулевой площади не выделяет ничего, нормализация
рамки, протянутой справа налево и снизу вверх. `CrossSection.test.tsx` — хром группы и счётчик при
переданном выделении. `LadeplanScreen.test.tsx` — группа уходит в буфер, счётчики склада сходятся.

Пиксельные жесты не тестируем: jsdom их не воспроизводит, и притворяться, что тест что-то проверил,
хуже, чем его отсутствие.

## Границы

Не входит: поворот группы; возврат группы из буфера (решено — поштучно); клавиатурный доступ к
выделению (`LKWkalk-dwc.5`); вид сбоку (он вообще не принимает правок); печать и PDF.
