# Умная раскладка — стратегии упаковки и плотность (LKWkalk-4bj.6)

> Спека T6 эпика `LKWkalk-4bj` (Ladeplan v2), 2026-07-16. Процесс: docs-first → TDD.
> Решения владельца получены письменно (см. §2). ADR: [016](../../adr/016-order-grouping-density-toggle.md),
> [017](../../adr/017-dense-floor-heuristic.md). Контракт: 0.9.0 → 0.10.0 (аддитивно).
> `4bj.6` — зонтичная задача; исполняются три подзадачи: `4bj.7`, `4bj.8`, `4bj.9`.

## 1. Контекст и постановка

Со скриншота прода: shelf-эвристика режет полки под первый тип груза → в полках остаются карманы
(`×16`/`×4`/`×7`), ~84% пола, часть груза уходит в `unplaced` при видимо свободном месте.

Разбор `packages/engine` (2026-07-16) уточняет постановку в двух местах:

**Во-первых, «режет полки под первый тип» — только один из трёх независимых источников потерь.**
Починка одного не сдвинет утилизацию на смешанных загрузках. Полный список — в
[ADR 017](../../adr/017-dense-floor-heuristic.md) §Контекст: храповик `shelfDepth` (`floor.ts:100`),
next-fit (`floor.ts:104-106`), ориентация против пустого региона (`floor.ts:88`). Четвёртый источник
— срезы зон заказов (`orchestrator.ts:88-126`) — ортогонален и разбирается в
[ADR 016](../../adr/016-order-grouping-density-toggle.md).

**Во-вторых, половина задачи уже реализована в движке и просто не выведена в UI:**
`Load.loadingMode: 'rear' | 'side' | 'combined'` ([ADR 012](../../adr/012-loading-modes.md),
контракт 0.4.0, дефолт `combined`) — это и есть «ряды по ширине / по длине». Работает, покрыт
тестами, недоступен пользователю. Это дёшево и отделено в `4bj.7`.

## 2. Решения владельца (2026-07-16)

| Вопрос | Решение |
|---|---|
| Набор стратегий в UI | Три режима сейчас (`4bj.7`); плотная эвристика — отдельной задачей (`4bj.8`) |
| Названия | «Automatisch» (combined) · «Von hinten laden» (rear) · «Von der Seite laden» (side) |
| Дефолтная стратегия | `combined` — контрактный дефолт ADR 012 сохраняется |
| Подача варианта | Авто-лучший + ручной переключатель (**не** галерея) |
| Смежность зон заказов | Тумблер «Dichte vor Auftragstrennung», по умолчанию **выкл** → [ADR 016](../../adr/016-order-grouping-density-toggle.md) |

## 3. Решения агента (дефолты приняты по директиве автономии)

- **«Maximale Dichte» не становится четвёртой стратегией в UI.** Плотная эвристика применяется
  внутри каждого режима. Карман внутри rear-полки — дефект укладки, а не свойство режима; выбор
  «плотно / неплотно» пользователю предлагать незачем. Побочный эффект — UI остаётся на трёх
  опциях, как и выбрано в §2. Обоснование и разбор влияния на свойство «остаток у доступного края»
  — [ADR 017](../../adr/017-dense-floor-heuristic.md).
- **Сорт футпринтов по убыванию площади (ADR 004) не реализуется.** Конфликт «сорт по площади vs
  порядок входа как приоритет» (`floor.test.ts:141`) — мнимый: best-fit и backfill дают плотность,
  **не переупорядочивая запросы**. Приоритет заявки сохраняется бесплатно. См.
  [ADR 017](../../adr/017-dense-floor-heuristic.md) §Решение.
- **Смена стратегии = новый расчёт → ручные правки (`editLayout.ts`) сбрасываются.** Сохранять их
  через пересчёт невозможно: правка привязана к координатам конкретной раскладки. UI предупреждает
  перед сбросом, если правки были.
- **Метрики сравнения — строкой под селектором, не галереей.** Расчёт в браузере и мгновенный, но
  вычислять все режимы всегда — лишняя работа; считаются лениво при раскрытии сравнения.

## 4. Объём по подзадачам

### 4.1 `4bj.7` — вывести `loadingMode` в UI (дёшево, контракт не меняется)

- Селектор стратегии на экране Ladeplan; значение → `Load.loadingMode`; пересчёт на месте.
- i18n de/ru: `ladeplan.strategy.*` (`automatic` / `rear` / `side` + подписи-пояснения).
- Персист: поле уже в `Load` → попадает в `localStorage 'ladungsplaner.load'` без новых ключей.
- Предупреждение о сбросе ручных правок при смене стратегии.
- **Не входит:** галерея вариантов, четвёртая стратегия, правки движка.

### 4.2 `4bj.8` — плотная напольная эвристика (дорого, только `packages/engine`)

Реализует [ADR 017](../../adr/017-dense-floor-heuristic.md): best-fit вместо next-fit, backfill
карманов, ориентация против остаточного места, `break` → `continue`. Всё внутри сигнатуры
`packFloor(region, requests, opts) → FloorPlacement[]` (`floor.ts:124`), единственный вызов —
`orchestrator.ts:108`; оркестратор не меняется.

**Порядок TDD:** сначала падающий тест на карман (смешанная загрузка, где сегодня остаётся
`shelfDepth − growExtent`), затем эвристика. Property-тест `floor.test.ts:189` — основной страж;
поднять `numRuns`.

### 4.3 `4bj.9` — тумблер `orderGrouping` (контракт 0.10.0, зависит от `4bj.7` + `4bj.8`)

Реализует [ADR 016](../../adr/016-order-grouping-density-toggle.md):
`Load.orderGrouping?: 'strict' | 'densityFirst'`, default `'strict'`; UI-тумблер выкл по умолчанию.

**Намеренно последняя:** после `4bj.8` значительная часть карманов исчезает без ослабления
смежности. Если выигрыш `densityFirst` окажется мал — тумблер дешевле не строить, чем строить и
объяснять. **Перед реализацией — измерить выигрыш на реальном кейсе со скриншота и записать число
в бид.**

## 5. Инварианты (проверяются на каждом результате)

- `findGeometryViolations(load, layout) === []` — property-тестами и в UI после ручных правок.
- Правила `nesting` / `stacking` / `rotation` не нарушаются ради заполнения; упаковщик порождает
  только yaw-ориентации ([ADR 013](../../adr/013-rotation-mvp-yaw.md)).
- Расчёт детерминирован: одинаковый `Load` → deep-equal `Layout`. Все тай-брейки явные, RNG нет.
- **Эталонные числа держатся:** EUR 1200×800 → 34, Gitterbox 1240×835 → 20, `rotation:'none'` → 33.
- Вид сбоку не рушится: конвенция `rowY`/depth (`cutaway.ts:107`, больший `y` = ближний ряд) —
  UI-поле, движок его не знает; новая раскладка не меняет контракт `Placement`.
- Ручные правки (`editLayout.ts`: `moveStack`, `rotateStack`) продолжают работать на любом
  результате новой эвристики.
- При `orderGrouping: 'strict'` (дефолт) — раскладка **байт-в-байт как сегодня**, кроме изменений
  плотности из `4bj.8` (регресс-тест на срезы зон).

## 6. Границы

- Гибрид «разные зоны — разной осью полки» — вне MVP ([ADR 012](../../adr/012-loading-modes.md)).
- LIFO-очередность точек маршрута — вне MVP ([ADR 011](../../adr/011-order-grouping.md)).
- Переворот на грань (полные 6 ориентаций) — пост-MVP ([ADR 013](../../adr/013-rotation-mvp-yaw.md)).
- Учёт веса — вне MVP; `Load.objective` объявлен (`types.ts:63`) и **нигде не читается** — мёртвое
  поле, трогать в рамках T6 не требуется.
- Оптимальность не заявляется: эвристика, а не точное решение 3D bin packing.
