# HANDOVER 2026-07-22 — релиз закреплён: continuous deployment из `main`

## Что сделано: `LKWkalk-7jg` (PR #15, `96aa6f0`) — закрыт

Прошлая сессия вскрыла системную причину: прод отставал от `main` на 77 коммитов, потому что
шаг «выложить на прод» нигде не был описан как часть процесса. Эта сессия его закрепила —
переходом на **continuous deployment из `main`**.

Решение (утверждено владельцем в брейншторме, [ADR 023](adr/023-continuous-deploy-from-main.md)):
- Coolify собирает ветку **`main`** (не `production`); мерж в `main` = выкладка на прод.
- Ветка `production` **ретайрнута** — ветки, которую можно забыть синхронизировать, больше нет.
- Инвариант: `main` всегда прод-готов; **шаг релиза = мерж в `main` + проверка live**.
- Гейт качества — **CI на GitHub Actions** (`.github/workflows/ci.yml`, job `ci`:
  build → typecheck → lint → test, Node 22), обязательный статус-чек на `main`.

Развилки, зафиксированные владельцем: релиз — автоматически после каждого мержа (не по команде,
не полу-авто); docs-only мержи тоже пересобирают прод (принято ради простоты — ноль path-фильтров).

## Главное открытие сессии: CD-модель безопасна ровно настолько, насколько зелёный `main` — и CI это доказал на первом прогоне

CI на первом же запуске **упал** и поймал реальный баг, который все локальные гейты и все три
раунда ревью пропустили: сборка `apps/web` падала на `tsc` — `TS7016`, нет `.d.ts` для
`@shadrin-v/contracts`.

**Причина.** Корневой `npm run build` = `npm run build --workspaces`. npm резолвит воркспейсы
**по алфавиту**: `contracts` собирается раньше `engine`, от которого зависит. На чистой сборке
`tsup --dts` для contracts отрабатывает без типов engine и выдаёт `index.js` без годного
`index.d.ts` → web не находит декларации.

**Почему молчали локальные гейты и ревью.** На машине разработчика `contracts/dist/*.d.ts` уже
лежал с прошлых сборок (dist в `.gitignore`, но физически на диске). Ревьюеры код не собирали из
чистого состояния. Тот же класс, что урок прошлой сессии про `packages/i18n` dist.

**Как обходит прод.** Dockerfile собирает пакеты в явном порядке
`i18n → engine → contracts → web → server`. CI теперь делает так же (`dbb75b0`), проверено из
чистого `rm -rf */dist`. Корневой скрипт остаётся кривым — заведён **`LKWkalk-dsu` (P3)**.

## Грабли (к прошлым в дополнение)

1. **Зелёный локальный гейт ≠ зелёная сборка из чистого состояния.** Gitignored `dist/` на диске
   маскирует ошибки порядка сборки монорепо. CI из чистого клона — единственная честная проверка.
   Прежде чем верить «build passes», собери из `rm -rf */dist`.
2. **`npm run build --workspaces` резолвит по алфавиту, а не по зависимостям.** Если пакет A
   зависит от B и A алфавитно раньше B — на чистой сборке A получит неполные типы. Явный порядок
   (как в Dockerfile) обязателен, пока `LKWkalk-dsu` не починит корневой скрипт.
3. **CD-модель разворачивает цену каждого пуша.** Теперь ЛЮБОЙ пуш в `main` (включая docs и
   `chore(beads)`) пересобирает прод ~4 мин. Это осознанный выбор владельца (простота вместо
   path-фильтров). Помни об этом: мелкие bookkeeping-коммиты тоже дёргают прод.
4. **Прямой пуш в `main` проходит с admin-bypass**, но branch protection при этом печатает
   `Required status check "ci" is expected` — это не ошибка, а подтверждение, что защита включена.
   Обычная работа — через PR (там `ci` обязателен), прямой пуш — только аварийно.
