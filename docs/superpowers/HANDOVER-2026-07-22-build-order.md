## Что сделано: `LKWkalk-dsu` (PR #16, squash `bba84d2`) — закрыт

Прошлая сессия ввела CD из `main` и на первом же прогоне CI поймала баг порядка сборки монорепо
(корневой `npm run build` собирал воркспейсы по алфавиту). Тогда его обошли явными шагами в
`ci.yml`/`Dockerfile` и завели follow-up `LKWkalk-dsu`. Эта сессия его закрыла.

**Причина.** Корневой `build` = `npm run build --workspaces`. npm резолвит воркспейсы **по алфавиту**:
`@shadrin-v/contracts` собирается раньше `@shadrin-v/engine`, от которого зависит. На чистом чекауте
`tsup --dts` для contracts отрабатывает без типов engine → `index.js` без годного `index.d.ts` →
`apps/web` (`tsc --noEmit`) падает на `TS7016`. Локально маскировалось устаревшим gitignored `dist/`.

**Фикс (3 правки + guard-тест):**
- `package.json` — `build` собирает воркспейсы в явном порядке зависимостей
  (`i18n → engine → contracts → web → server`) через один скрипт, вместо алфавитного `--workspaces`.
- `.github/workflows/ci.yml` и `Dockerfile` — те же 5 шагов, продублированных в трёх местах,
  схлопнуты в один `npm run build`. **Порядок сборки теперь живёт в одном месте** (это и был класс
  исходного бага — три копии, которые разъезжаются).
- `tests/build-order.test.ts` (+ `tests/**` glob в `vitest.config.ts`) — регрессионный инвариант:
  падает, если корневой скрипт откатится к `--workspaces`, перестанет покрывать воркспейс с `build`
  или нарушит порядок зависимостей. Проверен red на старом скрипте, green на фиксе.

**Проверка (systematic-debugging → TDD → review → PR → CD):**
- Падение воспроизведено из `rm -rf */dist` со старым скриптом (`TS7016`), фикс — green из чистого.
- **CI на PR #16 (сборка из чистого клона) — pass за 56 с.** Это и была честная кросс-проверка.
- Локально: typecheck 0 · lint 0 · **577 тестов** (было 574 + 3 новых).
- **Прод: healthy весь редеплой** — 10/10 замеров `GET /api/health` = `200` / `contract 0.14.0`
  сквозь окно пересборки. Новая Docker-сборка на `npm run build` поднялась.

## Значение для процесса: первый полный CD-цикл от фикса до прода — прошёл чисто

Ветка → зелёные гейты → PR → CI-гейт из чистого клона → squash-мерж в `main` = авто-деплой →
прод-верификация. Модель CD из `main` (ADR 023) отработала как задумано: расхождение прод/`main`
невозможно by design, а зелёный CI из чистого клона — единственная честная проверка сборки.

## Грабли (в дополнение к прошлым)

1. **`package.json` — строгий JSON, комментарий не вставить.** Флаг ревью «ручной список воркспейсов
   может молча пропустить новый пакет» закрыт не комментарием, а **тестом-инвариантом**
   (`tests/build-order.test.ts`). Добавил пакет — тест заставит вписать его в корневой `build`.
2. **`npm run build --workspace A --workspace B` выполняет в порядке перечисления** (не по алфавиту,
   как `--workspaces`). На этом держится фикс; порядок эмпирически подтверждён логом сборки.
3. **CD дёргает прод на любой пуш в `main`.** Этот handover-коммит (docs + git-мост beads) тоже
   пересоберёт прод ~4 мин. Осознанный выбор владельца (простота вместо path-фильтров).
4. **`codex login` не пройти из агента** (`stdin is not a terminal`). Кросс-модель `/codex` требует
   интерактивного логина владельцем в обычном терминале. Для dsu кросс-модель не требовалась
   (build-config, не БД/RLS/типы/критичный код) — обошлись CI из чистого клона.
