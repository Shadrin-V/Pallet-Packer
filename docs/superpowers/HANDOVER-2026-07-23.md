# HANDOVER 2026-07-23 — 41e.1: узнаваемый грузовик в разрезах Ladeplan (из вектора владельца)

## Что сделано: `LKWkalk-41e.1` — закрыт (ветка `feat/41e1-truck-chrome`, ещё НЕ в `main`)

Разрезы Ladeplan (вид сбоку **и** сверху) теперь читаются как **Sattelzug**: аэро-тягач, тридем-тележка,
опорные ноги, боковой противоподкатной брус, задний брызговик; сверху — кабина с зеркалами и сцепкой
(пятое колесо) + задние фитинги. Плюс **масштабная линейка в метрах** и диаграммы **во всю ширину**.

**Ключевое:** графика — из **ВЕКТОРНОГО РЕФЕРЕНСА ВЛАДЕЛЬЦА**
(`docs/design/Reference/truck_cargo_focus_top_side_aligned.svg` + `.eps`), а НЕ нарисована руками.
Тягач (side) и cab-top (top) вырезаны из него, перекрашены `#111`→`currentColor` (токен `--truck`),
привязаны к коробу в системе координат референса и масштабируются под габариты. **Груз остаётся точным
1:1**, грузовик не искажается ни на 13.6 м, ни на 6 м. Движок/контракт **не тронуты** — чистая
презентация в `apps/web` (+ i18n-ключ `ladeplan.rulerUnit`, токен `--truck: #3f4a44`).

`LKWkalk-51m` (реалистичный ассет) **свёрнут сюда** и закрыт.

## Решения / поздние фиксы

1. **Метод: только исходный вектор владельца.** Рукописный/CC0/трассировка — НЕ применять для
   реалистичных иллюстраций (см. Грабли). Механизм привязки к коробу — в `truckChrome.tsx`.
2. **Поздний фикс из финального ревью (opus):** метки линейки **обрезались** на длинных прицепах —
   `font = length·0.02` растёт с длиной, а лейн `rulerGutter` был `height·0.16` (растёт с высотой);
   при `length/height > ~4.2` (все стандартные тягачи) низ цифр резался внешним svg. Починено:
   `rulerGutter = max(height·0.16, length·0.045)` (`CrossSection.tsx`). **jsdom это не ловит** (нет
   layout) — проверено в браузере на LKW Standard 13600×2450 (ratio 5.55) → метки целые.
3. `--truck` затемнён `#5b6b62`→`#3f4a44` (тёмный slate-green для лайн-арта).
4. Убраны мёртвые константы `TOP`/`T_REAR`; top-ассет добавлен в hygiene-тест.

## Грабли

1. **Рукописный SVG-иллюстратор = тупик.** 3× «ужасно» (CC0-кабина, потом несколько рукописных
   лайн-арт-кабин, каждая рендерилась-и-осматривалась через headless-Chrome). Даже render→Read-луп не
   превращает подогнанные вручную пути в качество референса. **Сработало:** попросить у владельца
   ИСХОДНЫЙ ВЕКТОР и привязать его части к коробу. Урок в памяти `hand-svg-illustration-trap` +
   `bd remember 41e1-truck-chrome-source-vector`. Правило: нужна конкретная иллюстрация — **сразу
   проси исходный вектор пользователя**, не жги итерации.
2. **headless-Chrome render→Read — правильный инструмент, чтобы ВИДЕТЬ свой вывод** (не рисовать
   вслепую), но не заменяет качественный ассет. Пайплайн: SVG → `chrome --headless --screenshot` → Read PNG.
3. **jsdom не проверяет геометрию/клиппинг** — визуальные баги (клиппинг линейки) ловятся только
   настоящим Chrome. 315/315 apps/web зелёные НЕ гарантируют визуал.
4. **Локально `apps/server`-тесты красные** — `better-sqlite3 NODE_MODULE_VERSION 137` (нативный
   модуль под другую версию Node). Не связано с 41e.1; в CI (чистый Node 22) пересобирается и зелёное.
   Не пытаться «чинить» это в рамках фичи.
