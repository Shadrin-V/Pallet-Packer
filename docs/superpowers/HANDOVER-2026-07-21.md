# HANDOVER 2026-07-21 — групповое выделение стопок + удаление из расчёта

Обе задачи смержены в `main`. Ветки удалены, беды закрыты и синхронизированы (`bd dolt push`).

## Что сделано

### `LKWkalk-dwc.6` — групповое выделение и перенос стопок (PR #9, `619e463`)

Рамкой мыши по виду сверху выделяется несколько стопок; они переносятся как жёсткий блок или уходят
в буфер одним движением.

- Контракт движка `0.14.0` + [ADR 021](../adr/021-group-layout-edits.md): `unplaceStacks`,
  `moveStacks` (принимает **дельту**, а не целевые координаты — жёсткость по построению),
  `resolveGroupDrop` (групповой магнит, наследует правило «вплотную важнее, чем близко» из ADR 020).
- Отказ — всегда целиком, исходная раскладка возвращается по идентичности.
- Буфер асимметричен: туда группой, обратно поштучно (буфер не хранит форму группы).
- Поворот группы намеренно вне объёма: ручка поворота только при выделении одной стопки.

### `LKWkalk-yxn` + удаление из расчёта (PR #10, `7aa4560`)

- Позицию и заказ можно убрать из расчёта: корзина → «Подтвердить удаление». Взведена всегда одна
  кнопка; разоружение по Escape, нажатию вне и таймауту 4 с ([ADR 022](../adr/022-article-name-provenance-and-confirm-patterns.md)).
- Каталог при этом не трогается. Приложение писать в ERPNext **физически не умеет** — адаптер ходит
  наружу только GET-ом (`apps/server/src/erpnext/adapter.ts`).
- Экран настройки никогда не бывает пустым: заказ без позиций схлопывается, последний заменяется чистым.
- Имя артикула вошло в провенанс ERPNext: `upsertArticle` не меняет имя ERP-артикула, форма
  объясняет, что имя правится в ERPNext. Локальный артикул переименовывается как раньше.
- `ArticleConstructiveField` → `ArticleErpField`, `ARTICLE_CONSTRUCTIVE_FIELDS` → `ARTICLE_ERP_FIELDS`.

Состояние `main`: **573 теста**, typecheck и lint чистые.

## Решения владельца — помнить

- Имя артикула правится **исключительно в ERPNext**. Создание, удаление и переименование артикулов
  в ERP — только в ERP.
- Удаление из расчёта — это «не грузим в этот рейс», а не «такого артикула нет». Каталог не трогаем.
- Конструктивное поле заперто **тогда и только тогда**, когда ERPNext его фактически прислал
  (`erp_fields_json`), а НЕ когда «запись из ERP и значение непустое». Чинилось четыре раза.
- Поле имени остаётся редактируемым: это поисковая строка комбобокса.
- Автопоезд в пресеты не вносим, пока нет модели нескольких отсеков (`LKWkalk-p3p`).

## Грабли этой сессии — не наступать снова

1. **jsdom врёт про pointer-события.** Кнопка «взвод → подтверждение» не работала в настоящем Chrome
   при 573 зелёных тестах: React 18 применяет обновление состояния синхронно, кликнутый узел успевает
   открепиться, и `closest()` на нём возвращает `null`. В jsdom этого не видно, потому что `act()`
   откладывает перерисовку. **Любую жестовую механику проверять в браузере.**
2. **`docs/api-contract.md` — это контракт ДВИЖКА.** Его версия обязана совпадать с
   `ENGINE_CONTRACT_VERSION`. DTO каталога живут в `packages/contracts` и описаны в
   `docs/superpowers/specs/2026-07-20-article-autocomplete-design.md`. Не бампать движок за DTO.
3. **Тесты могут не отличать правильное правило от запрещённого.** Мутационная проверка показала, что
   `source==='erp' && имя непустое` проходило все 45 серверных тестов. Просить ревьюеров саботировать код.
4. **Правка может внести новую регрессию.** Исправление дублей номеров заказов сломало палитру.
5. Между `git push` и `gh pr merge` **сверять головы** — GitHub может смержить по устаревшей.
