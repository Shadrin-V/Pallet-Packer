# ADR 013 — Вращение в MVP: `full` ≈ yaw в упаковщике (переворот отложен)

Статус: Принято, дополнен ([ADR 018](018-fork-access-orientation.md)) · Дата: 2026-07-10

> **Дополнение 2026-07-16 ([ADR 018](018-fork-access-orientation.md)):** `rotation: 'full'`
> убирается из UI (в упаковщике и так ≈ yaw; в типе `RotationRule` остаётся для будущего). Ось yaw
> дополнительно ограничивается доступом погрузчика (`forkAccess`).

## Контекст

`RotationRule` = `none | yawOnly | full` ([контракт](../api-contract.md)). Номинально `full` — 6
ориентаций, включая переворот единицы на грань (одна из `lhw/hlw/whl/hwl`), где по вертикали
оказывается не высота, а длина или ширина.

Модель расчёта 2.5D ([ADR 003](003-2p5d-computation-model.md)) и вертикальный расчёт (`LKWkalk-qrd.5`)
оперируют **фиксированной высотой единицы H**. Переворот на грань делает вертикальный размер
переменным — вертикальная модель этого не умеет. В брейншторме `LKWkalk-qrd.4` переворот на грань уже
отложен как вне MVP; упаковщик кладёт только напольные (yaw) ориентации.

Логика «режим → ориентации» при этом дублировалась в трёх местах: `geometry.ts`
(`allowedOrientations`), `validate.ts` (`orientationTriples`), `floor.ts` (`chooseOrientation`).

## Решение

- **Упаковщик MVP использует только yaw-ориентации** для всех режимов: `none` → `lwh`;
  `yawOnly`/`full` → `{lwh, wlh}`. Четыре tipping-ориентации `full` для размещения **не используются**.
  Высота H единицы неизменна — 2.5D сохранён.
- **Валидация лояльна.** Проверка «влезает ли единица в кузов» (`fitsInVehicle` →
  `ERR_CARGO_EXCEEDS_VEHICLE`) для `full` учитывает все 6 ориентаций: `full`-груз, влезающий только
  перевёрнутым на грань, **не отклоняется** валидатором. Но упаковщик кладёт yaw → такая единица
  попадает в `unplaced` **без ошибки** (задокументированное ограничение MVP).
- **Единый источник маппинга.** Маппинг «режим → ориентации» и ось-маппинг выносятся в
  `packages/engine/src/model/orientation.ts` (`allowedOrientations`, `floorOrientations`,
  `orientedDims`), который потребляют geometry, validation и packer. Три копии → одна.
- **Контракт не меняется.** `RotationRule` и `Orientation` те же; `ORIENTATIONS` сохраняет все 6 для
  будущего.

## Последствия

- В MVP `full` поведенчески эквивалентен `yawOnly` на стадии укладки; расхождение только в лояльности
  валидатора (он признаёт 6 ориентаций `full` при проверке габаритов).
- Настоящий переворот на грань (полные 6 ориентаций в упаковке) — **пост-MVP задача**, требует, чтобы
  вертикальная модель (`LKWkalk-qrd.5`) считала высоту по фактической ориентации.
- DRY: устранено тройное дублирование rotation-логики; риск расхождения снят.
- Затрагивает `geometry.ts`, `validate.ts`, `floor.ts` (рефактор без смены поведения) и добавляет
  `model/orientation.ts`. Инвариант «недопустимые ориентации не используются» держится:
  упаковщик порождает только `lwh`/`wlh`, оба допустимы при `yawOnly`/`full`, а при `none` — только `lwh`.
