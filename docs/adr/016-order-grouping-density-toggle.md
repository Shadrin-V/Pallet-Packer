# ADR 016 — Ослабление смежности зон заказов: `orderGrouping: strict | densityFirst`

Статус: Принято · Дата: 2026-07-16 · Дополняет [ADR 011](011-order-grouping.md)

## Контекст

[ADR 011](011-order-grouping.md) требует: позиции одного `orderId` размещаются **смежной зоной**,
чтобы заказ выгружался целиком, не разбирая чужие поддоны. ADR 011 §Последствия это признаёт
платным: «смежность зоны заказа может оставлять пустоты — приемлемо ради выгрузки целиком».

В реализации смежность обеспечена **структурно**, а не проверкой: зоны — последовательные срезы по
длине кузова (`orchestrator.ts:88-126`), каждая зона получает регион
`[xOffset, vehicle.length] × width`, после зоны `xOffset += maxX + clearance`. Зона физически не
может писать в чужой срез, и **возврата в предыдущую зону нет**: незаполненный хвост последней полки
зоны становится мёртвым пространством, отнесённым на старт следующей зоны.

На мультизаказных загрузках это один из трёх источников потерь пола (два других — храповик
`shelfDepth` и next-fit — снимаются в [ADR 017](017-dense-floor-heuristic.md) **внутри** зоны, не
трогая смежность).

Владелец (2026-07-16) на вопрос «допустимо ли ради плотности ломать смежность зон» выбрал:
смежность по умолчанию жёсткая, но должна быть **явная возможность** её отключить. Это продуктовый
компромисс (влияет на работу грузчика на разгрузке), а не техническая деталь, поэтому фиксируется ADR.

## Решение

Вводится `Load.orderGrouping?: 'strict' | 'densityFirst'` (аддитивно, контракт **0.10.0**;
default `'strict'`).

- **`strict` (default):** поведение ADR 011 без изменений — зоны заказов смежны, каждая зона
  занимает свой срез по длине кузова. Отсутствие поля = `strict`, поэтому **сохранённые планы и
  существующие вызовы считаются байт-в-байт как сегодня**.
- **`densityFirst`:** зонирование не применяется — упаковщик получает единый регион на весь кузов и
  все позиции сразу; `orderId` перестаёт быть ограничением. Заказ может оказаться разбитым по кузову.

`orderId` при `densityFirst` **не исчезает из модели**: он остаётся для окраски/легенды/отчёта
(`Legend.tsx`, `orderBreakdown.ts`) — меняется только то, что он больше не ограничивает раскладку.

В UI — тумблер «Dichte vor Auftragstrennung», **по умолчанию выключен**, с честным объяснением
последствия («Aufträge müssen beim Entladen sortiert werden»), а не нейтральной формулировкой:
цена решения платится не тем, кто его принимает, а грузчиком на разгрузке.

## Последствия

- Контракт минорно растёт до 0.10.0 (только добавление `orderGrouping`); обратная совместимость
  полная (отсутствие поля = `strict`).
- ADR 011 **остаётся в силе как дефолт** — ослабление возможно только явным выбором владельца.
  Инвариант «заказ выгружается целиком» перестаёт быть безусловным инвариантом системы и становится
  свойством режима `strict`; формулировки, опирающиеся на его безусловность, поправлены.
- `zonesOf` (`orchestrator.ts:60`) при `densityFirst` схлопывается в одну неявную зону — то есть
  ветка «без `orderId` — единая неявная группа» (ADR 011) уже существует и переиспользуется;
  реализация — выбор источника зон, а не второй упаковщик.
- Порядок зон между собой остаётся нестрогим, LIFO-очередность точек маршрута — по-прежнему вне MVP
  ([ADR 011](011-order-grouping.md)). При `densityFirst` вопрос очередности выгрузки теряет смысл
  в принципе.
- Ожидаемый выигрыш `densityFirst` **не оценён количественно и может оказаться мал**: после
  [ADR 017](017-dense-floor-heuristic.md) (плотность внутри зоны) значительная часть карманов
  исчезает без ослабления смежности. Поэтому задача тумблера (`LKWkalk-4bj.9`) намеренно поставлена
  **после** плотной эвристики (`LKWkalk-4bj.8`): возможно, тумблер не понадобится вовсе, и тогда
  дешевле его не строить, чем строить и объяснять.
- Детерминизм сохраняется в обоих режимах; `findGeometryViolations === []` — инвариант обоих.
