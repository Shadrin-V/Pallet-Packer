# ADR 014 — Геометрия вложенных/штабельных колонок (валидатор column-aware)

Статус: Принято · Дата: 2026-07-10

## Контекст

Контракт моделирует раскладку по единицам: `Placement` на каждую единицу с полем `tier` (ярус).
Колонка на одном напольном месте — это стопка из `count` единиц (вертикальный расчёт `LKWkalk-qrd.5`).

- **Entschachtelt** (штабель): единицы на высотах `z = i·H`, каждая высотой `H` — полуоткрытые
  интервалы `[i·H, (i+1)·H)` соприкасаются, но НЕ пересекаются. Текущий валидатор геометрии
  (`LKWkalk-qrd.9`) их пропускает.
- **Verschachtelt** (вложение): единицы вкладываются друг в друга (в этом суть — «замок»), их
  bounding-box'ы физически ПЕРЕКРЫВАЮТСЯ (прирост пары `< H`). Инвариант spec «ни одна единица не
  пересекается с другой» вступает в противоречие с физикой вложения.

Оркестратор (`LKWkalk-qrd.7`) обязан прогонять валидатор геометрии на КАЖДОМ результате; значит
результат должен проходить проверку, не жертвуя моделью «единица = Placement».

## Решение

Инвариант «нет пересечений» уточняется: он применяется **между разными напольными колонками**, а не
внутри одной. Валидатор геометрии становится **column-aware**:

- Вертикальное перекрытие ДОПУСТИМО только для двух placement'ов, у которых **совпадают `(x, y)` и
  `cargoTypeId`** — это одна вложенная/штабельная колонка одного типа.
- Любое другое перекрытие — разных колонок, разных типов, или горизонтальное (`x`/`y`) —
  по-прежнему нарушение (`overlap`).
- Проверки выхода за габариты (`out-of-bounds`) и запрещённой ориентации не меняются.

Контракт (форма `Placement`/`Layout`) НЕ меняется. Форма входа/выхода стабильна.

## Последствия

- `LKWkalk-qrd.9` (`findGeometryViolations`) получает правило колонки: пара боксов с одинаковыми
  `(x, y, cargoTypeId)` не проверяется на пересечение между собой (это одна колонка).
- Инвариант в [spec.md](../spec.md) переформулирован: «единицы разных напольных колонок не
  пересекаются; выхода за габариты нет; вложение/штабель делят колонку по построению».
- Оркестратор эмитит `count` placement'ов на колонку (per-tier), верхняя единица укладывается в
  `Hк` (гарантирует вертикальный расчёт). Виз (вид сбоку, `LKWkalk-qrd.14`) получает по-единичную
  раскладку.
- Property-тесты упаковки проверяют инвариант с учётом колонок.
