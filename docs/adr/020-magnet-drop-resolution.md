# ADR 020 — Магнит постановки: поиск места в ядре, строгие операции остаются строгими

- **Статус:** принято (2026-07-17)
- **Контекст задачи:** эпик `LKWkalk-a5b` (склад 1:1 и магнит), задача `crb`
- **Контракт:** `0.13.0` (аддитивно)

## Контекст

Владелец: *«нужно проверить, чтобы при переносе, если есть место, ставка происходила без проблем,
даже если пользователь немножко ошибается в позиционировании»* — и, уточняя: *«можно наглядно
показать — подсветить красным границы наложения и зелёным как только вписывается»*.

Сегодня `placeStack` отказывает при пересечении хоть на миллиметр ([ADR 019](019-manual-layout-editing-api.md)).
Курсор держит стопку за середину, угол прижимается к габариту кузова — но соседей это не учитывает.
Пользователь целится в щель между стопками и получает красную строку вместо стопки. Обратной связи
до отпускания нет: об отказе узнаёшь постфактум.

Три вопроса требовали ответа.

**Где живёт поиск места?** Соблазн — решить в UI: там курсор, там пиксели. Но «куда стопка имеет право
встать» — это правила упаковки (габарит, пересечение, ориентация, доступ вил), то есть домен. UI,
который их знает, — вторая копия правил в заменяемом слое, ровно то, что запрещает принцип №1
(CLAUDE.md) и что ADR 019 только что вычистил.

**Флаг у `placeStack` или отдельная функция?** `tolerance?: number` у `placeStack` выглядит дешевле.

**Потянем ли live-подсветку?** Призрак надо пересчитывать на каждом `pointermove`, а `placeStack`
внутри зовёт `findGeometryViolations` — O(n²) по размещениям. При 700 паллетах на кадр это неприемлемо.

## Решение

**1. Магнит — чистая функция `resolveDrop` в ядре**, рядом с алгеброй правок:

```ts
resolveDrop(load, layout, spec, { tolerance?, exclude? }) → {
  x, y,                 // куда реально ляжет
  ok,                   // вписывается?
  error?,               // почему нет
  blocking: StackRef[]  // кто мешает — для красной подсветки
}
```

Кандидаты по каждой оси: прицел; обе стенки; **впритык** к кромкам каждого соседа. Фильтр — внутри
кузова и в пределах `tolerance` от прицела (по умолчанию половина меньшей стороны стопки). Выбор:
**больше «впритык»-осей → ближе к прицелу → `(x, y)` по возрастанию**.

**2. `placeStack`/`moveStack` остаются строгими.** Магнит — отдельная операция, не флаг.

**3. Позиционно-независимые правила проверяются однократно** — и это же делает подсветку дешёвой.

## Обоснования

**Почему прижим побеждает близость.** Погрузчик ставит паллеты кромка в кромку. Щель в 60 мм никому не
нужна и вдобавок врёт про плотность: она съедает место, которого потом «не хватает» следующей стопке.
Поэтому валидный прицел, оставляющий щель, всё равно притягивается вплотную — «ближайшее свободное
место» без прижима давало бы аккуратный на вид, но рыхлый результат.

**Почему отдельная функция, а не `tolerance` у `placeStack`.** `placeStack` судит **ту точку, которую
ей дали**. Вызывающая сторона по API/MCP должна иметь возможность сказать «поставь ровно сюда, иначе
откажи» — `placeStack`, который тихо переносит груз в другое место, это уже не та операция. Смешав их
флагом, мы бы отняли у контракта строгий примитив ради удобства одного жеста мышью. UI зовёт
`resolveDrop` → затем `placeStack` по найденной точке: два честных шага вместо одного двусмысленного.

**Почему live-подсветка точна и дешева.** `findGeometryViolations` проверяет четыре вида нарушений, но
**от координат зависят только два**:

| Нарушение | Зависит от `x`/`y`? | Цена |
|---|---|---|
| `orientation` | нет — только `rotation` типа | O(1), однократно |
| `fork-access` | нет — ориентация, `loadingMode`, `forkAxis` | O(1), однократно |
| `out-of-bounds` | да | O(1) |
| `overlap` | да | O(n) |

Отсюда устройство: ориентация и доступ вил проверяются **до** поиска — при них ни одна позиция не
годится, искать незачем. Остаётся O(кандидаты × n) ≈ 12 × 700 ≈ 8 тысяч сравнений на кадр, дешевле
уже существующего `setState` на каждое движение. Троттлинг через rAF не нужен.

Главное следствие: подсветка и постановка судят **по одному и тому же правилу**, поэтому зелёный
призрак не может обернуться отказом. Тест это и сторожит перебором сетки прицелов.

## Границы

`resolveDrop` решает вопрос **позиции**. Она намеренно **не** проверяет наличие свободных единиц
(`ERR_EDIT_NOTHING_TO_PLACE`): это не вопрос позиции и вовсе бессмыслен при переносе уже стоящей
стопки. Единицы остаются на `placeStack`. Инвариант «подсветка не врёт» цел: UI тащит плитку, которая
в буфере уже лежит, — значит единицы есть по построению.

## Последствия

- Контракт **0.13.0**, аддитивно; `@shadrin-v/engine` → `0.0.8`. Существующие операции не тронуты.
- UI получил право показывать исход до отпускания — и обязан применять **ровно показанное**
  (`placeStack` по `preview.x/y`), иначе подсветка становится ложью.
- Появился второй потребитель `overlaps1d` и «коробок пола». Если добавится третий — вынести в общий
  геометрический модуль; на двух это было бы преждевременно.
- Магнит доступен будущему MCP как самостоятельный инструмент («куда это влезет?»), отдельно от
  «поставь сюда».
