# ADR 018 — Доступ погрузчика как ограничение ориентации (`forkAccess` + ось)

Статус: Принято · Дата: 2026-07-16 · Уточняет [ADR 013](013-rotation-mvp-yaw.md),
[ADR 017](017-dense-floor-heuristic.md)

## Контекст

Уточнено владельцем (2026-07-16): некоторые поддоны конструктивно берутся погрузчиком только с
**двух противоположных сторон** (2-way), другие — с любой из четырёх (4-way). Стопку, развёрнутую
недоступной стороной к двери, физически **нельзя снять с фуры** — это жёсткий физический инвариант, в
одном ряду с «не выходит за габариты» / «не пересекается», а не эстетика раскладки.

Текущая модель ориентации ([ADR 013](013-rotation-mvp-yaw.md)) знает только `rotation`
(`none | yawOnly | full`) — «можно ли вращать ради плотности». Она не выражает, **какие стороны
доступны погрузчику**, поэтому плотная эвристика ([ADR 017](017-dense-floor-heuristic.md)), выбирая
ориентацию «по макс-влезанию», может развернуть 2-way поддон в нерагружаемую сторону.

Доступ погрузчика связан с направлением загрузки ([ADR 012](012-loading-modes.md)): погрузчик
заходит со стороны доступной двери, и доступная пара сторон стопки должна смотреть **к этой двери**.

Владелец разрешил ради этого убрать из UI разворот во все 6 ориентаций (`full`) — в упаковщике он и
так эквивалентен yaw ([ADR 013](013-rotation-mvp-yaw.md)), так что цена мала.

## Решение

### Контракт (аддитивно, 0.11.0)

```ts
type ForkAccess = 'all4' | 'twoSides';       // доступ погрузчика к стопке
type ForkAxis   = 'length' | 'width';        // ось захода вил (для twoSides)

interface CargoType {
  // ...
  forkAccess?: ForkAccess;   // default 'all4' (текущее поведение, без ограничения)
  forkAxis?: ForkAxis;       // default 'length'; значим только при forkAccess:'twoSides'
}
```

Отсутствие `forkAccess` = `'all4'` → **обратная совместимость полная**: сохранённые планы и
существующие вызовы считаются как прежде.

**`forkAxis`** задаёт, вдоль какой оси **самого поддона** заходят вилы: `'length'` — вилы
параллельны длине поддона, доступные грани нормальны длине (типичный 2-way EUR: вилы с торцов).

### Геометрия ограничения

Вилы входят в грани, **нормальные** оси захода. Погрузчик у задней двери движется вдоль `x` (длина
кузова) → вилы параллельны `x` → доступная пара граней стопки должна быть нормальна `x`. Значит:

- **`loadingMode: 'rear'`** — ось захода поддона (`forkAxis`) должна лечь **вдоль `x`**.
- **`loadingMode: 'side'`** — вилы параллельны `y` → ось захода должна лечь **вдоль `y`**.
- **`loadingMode: 'combined'`** — доступны и дверь, и борт → годится любая из двух yaw-ориентаций →
  ограничение **не пиннит**, эвристика вращает свободно.

Для `forkAccess: 'all4'` доступны все стороны → ограничение не действует ни в одном режиме.

### Влияние на выбор ориентации

`chooseOrientation` ([ADR 017](017-dense-floor-heuristic.md), `floor.ts`) получает **жёсткий фильтр
перед макс-влезанием**: сперва отбрасываются yaw-ориентации, нарушающие доступ при текущем
`loadingMode`, затем среди выживших берётся плотнейшая (тай-брейк `lwh` сохраняется). Для 2-way
поддона в `rear`/`side` остаётся **одна** ориентация — она пиннится, плотность ей уступает (физика
важнее заполнения, доменный инвариант). Поэтому этот ADR реализуется **до** плотной эвристики
(`LKWkalk-4bj.8` зависит от `LKWkalk-4bj.10`).

Если пиннутая ориентация не влезает — единица уходит в `unplaced` **без ошибки** (как tipped-only
`full` в [ADR 013](013-rotation-mvp-yaw.md)); валидация вход не отклоняет.

### UI (один контрол вместо `rotation`)

На артикуле — один список «Ausrichtung», маппится в контрактные поля:

| UI-пункт                              | `rotation` | `forkAccess` | `forkAxis`     |
|---------------------------------------|------------|--------------|----------------|
| Fixiert (nicht drehen)                | `none`     | (нерелевантно) | —            |
| Frei drehbar — Zugriff 4 Seiten       | `yawOnly`  | `all4`       | —              |
| Drehbar — Zugriff nur 2 Seiten        | `yawOnly`  | `twoSides`   | length/width ↴ |

При выборе «2 Seiten» появляется под-контрол оси (Länge / Breite). `rotation: 'full'` из UI
**убирается** (в типе остаётся для будущего, [ADR 013](013-rotation-mvp-yaw.md)).

## Последствия

- Контракт минорно растёт до **0.11.0** (только новые опц. поля); дефолт `all4` = текущее поведение.
- Доменный инвариант расширяется: «стопка стоит доступной стороной к двери» для 2-way поддонов —
  наравне с габаритами и непересечением. Упаковщик его не нарушает никогда, даже ради плотности.
- **Порядок работ:** fork-access (`4bj.10`) → плотная эвристика (`4bj.8`). Иначе `chooseOrientation`
  переписывается дважды.
- **Ручные правки.** `rotateStack` (`editLayout.ts`, [ADR 013](013-rotation-mvp-yaw.md)) должен
  отклонять поворот, ломающий доступ 2-way поддона при `rear`/`side` (как уже отклоняет выход за
  габариты). Способ: либо явная проверка в `rotateStack`, либо новый вид нарушения
  `'fork-access'` в `findGeometryViolations` — у неё уже есть и `Load.loadingMode`, и
  `CargoType.forkAccess` (решается в TDD `4bj.10`; при выборе нарушения — это аддитивное значение
  enum `GeometryViolation.kind`).
- `combined` остаётся дефолтом загрузки и не пиннит — самый частый кейс не теряет плотность.
- Оптимальность по-прежнему не заявляется; ограничение только сужает допустимые ориентации.
