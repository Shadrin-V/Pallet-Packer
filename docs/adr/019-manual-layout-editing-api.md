# ADR 019 — Ручная правка раскладки: операции в ядре, причина отказа в контракте

- **Статус:** принято (2026-07-17)
- **Контекст задачи:** эпик `LKWkalk-dwc` (буфер стопок), задача `dwc.1`
- **Контракт:** `0.12.0` (аддитивно)

## Контекст

Владелец: *«похоже, раскладку нужно будет править часто вручную… если места не хватает — программа
не позволяет повернуть стопку, а затем если есть ещё не загруженные, которые теперь влазят, их
неоткуда взять»*. Утверждено решение — полоса-буфер под видом сверху с двусторонним переносом
(вынуть стопку из кузова → повернуть → вернуть; поставить неразмещённую).

Это требует двух новых операций поверх раскладки — **снять стопку с пола** и **поставить стопку** —
и упирается в два факта:

1. **Логика уже есть в ядре, но наружу не отдана.** Построение колонки (`columnPlacements`: правила
   вложения/штабелирования, `z` каждого яруса — ADR 003/009) и метрики (`computeFillMetrics`) —
   внутренние модули `packages/engine`.
2. **Правки живут в UI.** `apps/web/.../editLayout.ts` содержит `moveStack`/`rotateStack` —
   доменные по сути функции (валидация геометрии, правила ориентации), собранные из примитивов ядра.
   Это противоречит принципу №1 проекта: *«вся доменная логика (валидация, упаковка, метрики) — только
   в ядре, никогда в UI»* (CLAUDE.md).

Наивный путь — доэкспортировать `columnPlacements` и `computeFillMetrics` и дописать
`unplaceStack`/`placeStack` рядом в UI. Тогда UI начнёт **пересобирать колонку и пересчитывать
метрики сам**, то есть заведёт вторую копию правил упаковки в заменяемом слое.

## Решение

**Алгебра правок раскладки — операция ядра, а не UI.** В `packages/engine` появляется модуль правок;
контракт `0.12.0` добавляет пять операций (аддитивно, ничего не переименовано и не удалено):

| Операция        | Вход                                        | Выход        |
|-----------------|---------------------------------------------|--------------|
| `moveStack`     | `Load`, `Layout`, `StackRef`, `x`, `y`      | `EditResult` |
| `rotateStack`   | `Load`, `Layout`, `StackRef`                | `EditResult` |
| `unplaceStack`  | `Load`, `Layout`, `StackRef`                | `EditResult` |
| `placeStack`    | `Load`, `Layout`, `PlaceStackSpec`          | `EditResult` |
| `stackBuffer`   | `Load`, `Layout`                            | `BufferStack[]` |

`moveStack`/`rotateStack` **переезжают** из `apps/web` в ядро вместе со своими тестами; UI сохраняет
только то, что действительно его: указатель, snap-сетку и перетаскивание.

### `EditResult` — отказ с причиной, а не молчание

```ts
interface EditResult { layout: Layout; error?: EngineError }
```

Сегодня отказ — это «вернуть прежний layout», и UI не может объяснить пользователю, почему кнопка
поворота будто не работает (`dwc.4`). Теперь при отказе возвращается **исходный layout + код**:

| Код                        | Условие                                                         |
|----------------------------|-----------------------------------------------------------------|
| `ERR_EDIT_NO_STACK`        | По `StackRef` нет колонки                                        |
| `ERR_EDIT_OVERLAP`         | Футпринт пересекает другую стопку                                 |
| `ERR_EDIT_OUT_OF_BOUNDS`   | Стопка выходит за габарит кузова                                  |
| `ERR_EDIT_FORK_ACCESS`     | Ориентация ломает доступ погрузчика при текущем `loadingMode` (ADR 018) |
| `ERR_EDIT_ROTATION`        | Тип запрещает вращение (`rotation: 'none'`) или колонка не yaw-однородна |
| `ERR_EDIT_NOTHING_TO_PLACE`| Ставить нечего: у типа нет неразмещённых единиц                    |

Текст — на стороне UI (ADR 006), как и у остальных кодов.

### Инварианты (не ослабляются ручной правкой)

- Результат любой операции проходит `findGeometryViolations(load, layout) === []`; иначе — отказ и
  **исходный** layout. Правило `nesting`/`stacking`/`rotation`/`forkAccess` не нарушается ради удобства.
- `placeStack` строит колонку **тем же** `columnPlacements`, что и упаковщик: высота колонки —
  из `computeVerticalStack`, `z` ярусов — по ADR 003/009. Ручная стопка не может быть выше расчётной.
- `unplaceStack`/`placeStack` **сохраняют баланс**: снятые единицы уходят в `Layout.unplaced`,
  поставленные — вычитаются оттуда. `placed + unplaced` по типу неизменно.
- Метрики после каждой правки пересчитываются ядром (`computeFillMetrics`); UI их не считает.
- Детерминизм: одинаковый вход → одинаковый выход.

### `stackBuffer` — буфер как стопки, а не как штуки

`Layout.unplaced` — это счётчики единиц. Пользователь же таскает **стопку**. `stackBuffer` собирает
из счётчиков полные стопки по `computeVerticalStack(cargo, vehicle).count`, хвост отдаёт неполной
стопкой; порядок — порядок `cargo` в `Load` (детерминизм, приоритет заявки).

## Последствия

- **+** Правила упаковки остаются в одном месте; UI и будущий MCP получают одну и ту же алгебру
  правок. Операции ложатся в MCP-инструменты один-к-одному (принцип №4).
- **+** Отказ объясним: у UI есть код причины (`dwc.4` закрывается по построению).
- **+** `apps/web/.../editLayout.ts` худеет до адаптера (snap + вызов ядра).
- **−** Контракт растёт на пять операций и шесть кодов ошибок; версия `0.12.0`, пакет `@shadrin-v/engine`
  требует ≥ patch-бампа перед публикацией (см. `docs/CHANGELOG.md` «Соответствие версий»).
- **−** `Layout` теперь мутируется двумя источниками (упаковщик и правки) — поэтому баланс
  `placed + unplaced` вынесен в инвариант и покрыт тестами.
- Вне объёма: undo/redo (UI-состояние), multi-select, правка вертикали (яруса) — только целые колонки.
