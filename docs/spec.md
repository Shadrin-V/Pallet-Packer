# Спецификация — Pallet Packer (MVP)

> Что делает система. Источник истины по требованиям. Как устроено — см. [design.md](design.md);
> контракт движка — [api-contract.md](api-contract.md); решения — [adr/](adr/).
> Статус: утверждена в брейншторме (фаза 1). Версия MVP: 0.1.0.

## 1. Назначение

Инструмент рассчитывает раскладку поддонов при загрузке грузового автомобиля (LKW):
по заданному кузову и заявке (список типов поддонов с количествами) строит оптимальную
геометрическую раскладку, показывает, сколько единиц каждого типа помещается и что не влезло,
и визуализирует результат. Аналог назначения «Astra Раскрой», но для загрузки фуры поддонами.

MVP решает **чисто геометрическую** задачу (без веса, осей, порядка выгрузки).

## 2. Основной сценарий (режим «Размести заявку»)

1. Пользователь выбирает кузов (из пресетов или свой).
2. Добавляет одну или несколько позиций заявки — тип поддона с размерами, правилами и количеством
   (или флагом «заполнить остаток»).
3. Задаёт состояние вложенности (Verschachtelt/Entschachtelt) и зазор.
4. Получает: сколько единиц каждого типа размещено и сколько не влезло; раскладку
   (вид сверху + вид сбоку); процент заполнения; возможность экспорта отчёта.

Частый случай — один тип поддона (заявка из одной позиции). Разные типы в одной фуре
поддерживаются, но редки.

## 3. Модель предметной области

### 3.1 Vehicle (кузов)
- Внутренние размеры грузового отсека: `length × width × height` (целые миллиметры).
- Библиотека пресетов типовых фур + пользовательские варианты.
- `maxPayload` (кг) — опциональное поле, в MVP **не проверяется** (заложено под будущий учёт веса).

### 3.2 CargoType (тип поддона / груза)
- Размеры `length × width × height` (целые миллиметры).
- `quantity` — требуемое количество, либо `fill: true` («разместить как можно больше»).
- Правила размещения:
  - **rotation**: `none` (без вращения) / `yawOnly` (поворот вокруг вертикали, обмен L↔W)
    / `full` (все 6 ориентаций).
  - **stacking**: `stackable` (можно штабелировать полноразмерные) + опц. `maxTiers`.
  - **nesting**: `nestable` (единицы вкладываются) + `stepHeight` + опц. `maxNested` +
    `nestingMode` (`sequential` | `pairwise`, default `sequential`) + `allowUnpairedTop?`
    (default false, только pairwise). Значение `stepHeight`: `sequential` → Δh (прирост на
    вложение); `pairwise` → h_д (высота двух верхних досок пары). См. [ADR 009](adr/009-pairwise-nesting-model.md).
  - **state**: `verschachtelt` | `entschachtelt` — состояние вложенности **на уровне типа**
    (у разных типов может отличаться). В UI есть глобальный переключатель для частого случая.
- `weightPerUnit` (кг) — опциональное поле, в MVP не используется.

### 3.3 Load (запрос на расчёт)
- `vehicle`, `cargo: CargoType[]`, `clearance` (мм, по умолчанию 0), `objective`
  (в MVP — `maxUnits`).

### 3.4 Layout (результат)
- `placements[]`: для каждой размещённой единицы — `cargoTypeId`, координаты `x, y, z`
  (мм, от угла отсека), `orientation`, `tier` (ярус), `state`.
- `unplaced[]`: `cargoTypeId → количество не размещённых`.
- Метрики: `floorFillPercent`, `volumeFillPercent`, `usedFloorPositions`, `totalPlaced`.

## 4. Модель расчёта (2.5D)

1. **2D-раскладка по полу.** Футпринты поддонов (`length × width`, с учётом `clearance`
   и допустимых поворотов) размещаются на плоскости пола кузова → набор напольных мест.
2. **Вертикаль.** Каждое напольное место — однородная вертикальная стопка **одного типа**.
   Число единиц по высоте зависит от состояния:
   - **Entschachtelt**: `tiers = floor(H_кузова / H)`, ограничено `maxTiers`.
   - **Verschachtelt, `sequential`**: максимальное `n`, при котором `H + (n − 1)·Δh ≤ H_кузова`,
     ограничено `maxNested`.
   - **Verschachtelt, `pairwise`** (вложение парами, [ADR 009](adr/009-pairwise-nesting-model.md)):
     `k = floor((H_кузова − H)/(H + h_д))`, `n = 2k + 1`; при `allowUnpairedTop` и остатке ≥ H
     → `n + 1`; затем `n = min(n, maxNested)` (потолок); если `(n − 1)` нечётно и не
     `allowUnpairedTop` → `n − 1`. Структура: 1 нижний одиночный + `k` пар + [непарный сверху].
3. **Раскладка заявки.** Запрошенные количества распределяются по местам × ярусам,
   пока не исчерпаны или не закончилось место.

**Допущение MVP:** одно напольное место = стопка одного типа (типы не смешиваются по высоте).

## 5. Критерий оптимальности

- Основной: **максимум размещённых единиц** (`maxUnits`).
- Тай-брейк: максимальное заполнение пола, затем компактность.
- Результат **детерминирован** при одинаковом вводе.

Будущие критерии (вне MVP): максимум заполнения объёма, минимум машин (мульти-рейс).

## 6. Слой UI (Lovable)

- Первый интерфейс — прототип в **Lovable** (React), клиент движка; движок считает в браузере.
- UI — **заменяемый слой**: правки логики идут в ядро, правки интерфейса — в Lovable.
- Экраны MVP: выбор кузова, редактор заявки, результат с двумя проекциями (сверху/сбоку),
  экспорт.
- Ломающее изменение контракта движка → сначала ADR и правка [api-contract.md](api-contract.md),
  потом реализация.

## 7. Интернационализация (i18n)

- Локали MVP: **de, ru** (en — позже). Архитектура допускает добавление языков без правок кода.
- Ни одной пользовательской строки в коде — только ключи локалей.
- Локализуются: интерфейс, подписи отчёта, единицы измерения, форматы чисел.
- Движок текстов не хранит: ошибки возвращаются **кодами**, перевод — на стороне UI.

## 8. Хранение справочников

- Встроенные пресеты: типовые фуры и стандартные Европоддоны — конкретные значения в [Приложении A](#приложение-a--пресеты).
- Пользовательские кузова и типы поддонов — в IndexedDB браузера.
- Экспорт/импорт всей библиотеки одним JSON-файлом (бэкап/перенос).
- CSV/Excel-импорт заявки — вне MVP.

## 9. Визуализация и экспорт

- **Вид сверху**: план пола — прямоугольники поддонов, ориентация, нумерация мест.
- **Вид сбоку**: ярусы / вложение по высоте.
- Экспорт: **PDF-отчёт + PNG-картинка + JSON** (сырая раскладка).
- Визуализация проверяется валидатором геометрии, а не сравнением с эталонной картинкой.

## 10. MCP-готовность

API движка проектируется под будущую публикацию как MCP-сервер без рефакторинга ядра.
Операции = будущие инструменты; входы/выходы — JSON-схемы уже сейчас. Подробности и перечень —
[api-contract.md](api-contract.md), раздел «MCP-готовность». Реализация MCP-сервера — отдельный
будущий эпик, в MVP не входит.

## 11. Доменные инварианты

- Ни одна единица не выходит за габариты кузова и не пересекается с другой.
- Правила nesting/stacking/rotation типа не нарушаются никогда, даже ради лучшего заполнения.
- Высота вложенной стопки — по формуле активного `nestingMode`: `sequential` → `H + (n − 1)·Δh`;
  `pairwise` → `H + k·(H + h_д)` (`+ H` при непарном верхнем). Никогда не `n·H`.
- Ориентация — из разрешённых для типа; при `rotation: none` используется исходная.
- Результат детерминирован при одинаковом вводе.

## 12. Тестирование

- Тривиальные точные случаи: пол 2×2 при поддоне 1×1 → 4 места; проверка ярусов по высоте
  для обоих состояний.
- Property-based: на каждом результате — отсутствие пересечений и выхода за габариты.
- По кейсу на каждое правило: nesting, лимит ярусов, каждый режим rotation, оба состояния.
- Смешанные типы; кейс «ничего не помещается».
- Проверка раскладки — валидатором геометрии.

## 13. Вне объёма MVP (заложено, не реализовано)

Вес и распределение по осям; порядок выгрузки (LIFO); хрупкость и «этой стороной вверх»;
мульти-рейс; 3D-визуализация; CSV/Excel-импорт; MCP-сервер; PLP-оптимальные паттерны и
метаэвристики; локаль en; смешивание типов в одной вертикальной стопке.
