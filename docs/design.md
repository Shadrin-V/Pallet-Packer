# Дизайн-документ — Pallet Packer (MVP)

> Как устроена система. Что делает — см. [spec.md](spec.md); контракт — [api-contract.md](api-contract.md).
> Решения зафиксированы в [adr/](adr/).

## 1. Обзор

Система — headless-движок упаковки на TypeScript плюс заменяемый UI-клиент в Lovable.
Движок — чистый изоморфный пакет: принимает JSON-запрос (кузов + заявка + правила),
возвращает JSON-раскладку. UI импортирует пакет и вызывает его прямо в браузере.
Тот же пакет позже оборачивается в Node REST / MCP-сервер без изменений ядра
(см. [ADR 001](adr/001-headless-ts-engine-in-browser.md)).

## 2. Пакеты и границы

```
@shadrin-v/engine   ← чистый TS, без DOM/Node. Домен, валидация, упаковка, метрики.
@shadrin-v/i18n     ← словари локалей (de, ru), форматирование единиц и чисел.
UI (Lovable)     ← React-клиент: экраны, рендер раскладки, экспорт. Импортирует engine + i18n.

Будущее (вне MVP), поверх того же engine:
@shadrin-v/rest     ← Node HTTP-обёртка.
@shadrin-v/mcp      ← MCP-сервер (операции = инструменты).
```

Каждый пакет имеет одно назначение, общается через типизированный JSON-контракт и тестируется
независимо. Ядро не знает про UI; UI не содержит доменной логики.

## 3. Внутреннее устройство `@shadrin-v/engine`

### 3.1 Модули
- `model/` — типы домена (Vehicle, CargoType, Load, Layout, …), единицы (int мм).
- `validation/` — проверка входа, возврат кодов ошибок.
- `packing/`
  - `Packer` — интерфейс стратегии упаковки (сменяемый).
  - `floor/` — 2D-упаковка футпринтов по полу (shelf/skyline-эвристика).
  - `vertical/` — расчёт ярусов на место по состоянию (nesting/stacking).
  - `orchestrator` — связывает floor + vertical + распределение количеств заявки.
- `metrics/` — заполнение пола/объёма, счётчики.
- `report/` — сборка структуры отчёта (без текста, только данные + коды).
- `index.ts` — публичный API по контракту.

### 3.2 Интерфейс упаковки

```ts
interface Packer {
  pack(load: Load): Layout; // детерминирован; не мутирует вход
}
```

MVP-реализация — `ShelfPacker`:

1. **2D по полу.** Для каждого типа берём футпринт `L×W`; при `yawOnly`/`full` рассматриваем и
   поворот на 90° в плоскости пола. Прямоугольники сортируются по убыванию площади / длинной
   стороны (стабильно), размещаются по «полкам» (shelf) слева-направо, снизу-вверх, с учётом
   `clearance`. Для одинаковых поддонов это даёт ровные ряды. Результат — список напольных мест
   с координатой и ориентацией.
2. **Вертикаль.** Для каждого места по типу и состоянию считаем число единиц по высоте:
   - Entschachtelt: `floor(H_кузова / H)`, ≤ `maxTiers` (если `stackable`, иначе 1);
   - Verschachtelt: max `k` при `H + (k−1)·Δh ≤ H_кузова`, ≤ `maxNested`.
3. **Распределение заявки.** Идём по местам, наполняем ярусами, вычитая из требуемых количеств,
   пока не исчерпаны или не кончились места. Остаток → `unplaced`.

Стратегия за интерфейсом: позже добавляются `PlpPacker` (оптимальные паттерны для одного типа)
и метаэвристики (со фиксированным seed ради детерминизма) — без изменения оркестратора и UI.

### 3.3 Система координат
- Начало — угол грузового отсека. Оси: `x` — длина, `y` — ширина, `z` — высота. Единицы — мм (int).
- Ориентация единицы — перечисление разрешённых для типа поворотов; хранится в `placement.orientation`.

## 4. Поток данных

```
UI (Lovable)
  │  Load (JSON)
  ▼
@shadrin-v/engine.calculateLayout(load)
  ├─ validation → коды ошибок при неверном вводе
  ├─ Packer.pack → placements + unplaced
  ├─ metrics → проценты заполнения
  └─ Layout (JSON)
  ▼
UI: рендер (вид сверху / сбоку) + i18n + экспорт (PDF/PNG/JSON)
```

## 5. Обработка ошибок

- Движок валидирует вход и возвращает **коды** (`ERR_*`, см. [api-contract.md](api-contract.md)),
  без человекочитаемого текста. Перевод и показ — UI через `@shadrin-v/i18n`.
- Некорректная геометрия (груз больше кузова, неположительный размер и т.п.) → ошибка валидации,
  расчёт не запускается.
- «Ничего не влезло» — это не ошибка: валидная раскладка с пустым `placements` и полным `unplaced`.

## 6. Детерминизм

- Никакой стохастики в MVP. Все сортировки стабильны, порядок обхода фиксирован.
- Одинаковый вход → побайтово одинаковый выход. Это делает раскладку тестируемой и
  воспроизводимой; будущие стохастические стратегии обязаны принимать и фиксировать seed.

## 7. Тестируемость

- Ядро без побочных эффектов и внешних зависимостей — тестируется юнит- и property-based тестами.
- Валидатор геометрии (`assertNoOverlap`, `assertWithinBounds`, `assertRulesRespected`)
  применяется к каждому результату в тестах и может переиспользоваться как самопроверка движка
  в dev-режиме.

## 8. Технологический стек

- Ядро и i18n: **TypeScript** (изоморфный, target ES2020+), сборка в ESM npm-пакет.
- Тесты: раннер на выбор реализации (напр. Vitest) + property-based (напр. fast-check).
- UI: **Lovable/React**; хранение — IndexedDB; экспорт PDF/PNG — клиентские библиотеки.
- Точный выбор инструментов тестов/сборки фиксируется в первой задаче реализации.
