## 6. Публикация `@shadrin-v/engine` (публичный npm)

Пакет публикуется в **публичный npm** (ADR-010 пересмотрен: GitHub Packages отклонён; приватность
кода нерелевантна — движок бандлится в браузер). `.npmrc`/секреты **не нужны**.

### Кто и как публикует
- **Публикует только пользователь.** У npm-аккаунта (`shadrin-v`) включён **2FA через passkey**
  (подтверждение в браузере) — поэтому в инструкциях **НИКОГДА не писать `--otp=XXXXXX`**. Команда
  публикации всегда ровно такая:
  ```
  git switch main && git pull
  cd packages/engine && npm publish --access public
  ```
- **Бамп версии пакета — обязанность агента**, часть merge в main, НЕ пользователя. Каждая
  публикация = **новая версия**: npm-реестр не даёт перезаписать уже опубликованную версию.

### Правило версий (агент соблюдает при каждом merge)
- Изменился контракт (`ENGINE_CONTRACT_VERSION`: 0.6.0, 0.7.0, …) → **минимум patch-бамп** пакета
  (`packages/engine/package.json`). Одно контрактное изменение = один патч-бамп при merge.
- К релизу MVP пакет переходит на `0.1.0`, дальше — обычный семвер.
- Таблица соответствия **версия пакета ↔ версия контракта ↔ что вошло** ведётся в
  [CHANGELOG.md](CHANGELOG.md) (раздел «Соответствие версий»), чтобы по версии в Lovable всегда было
  ясно, какой контракт внутри.

### Батчинг публикаций
- НЕ просить публикацию под каждое изменение. Копить в main; запрашивать републиш **только когда
  очередной Lovable-промпт реально требует новой версии**.
- Один запрос = один блок стандартного формата:
  > Публикуй `X.Y.Z` (контракт `A.B.C`): [однострочно что внутри]. После публикации в Lovable:
  > `Update @shadrin-v/engine to exactly X.Y.Z`, затем **Промпт N**.
- Пользователь подтвердил версию в Lovable → агент **закрывает републиш-бид**.

**Проверка перед запросом:** `npm view @shadrin-v/engine versions` — если целевая версия уже
опубликована, бампнуть до следующего патча ДО запроса (перезапись невозможна).
