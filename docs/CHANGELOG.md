## [Unreleased]

### 2026-07-21 — каталог артикулов снова доступен в браузере (`LKWkalk-7wb`)

- Исправлено: ни один запрос к `/api/` из браузера не уходил, поэтому сохранённые артикулы никогда
  не предлагались, а видны были только встроенные пресеты поддонов. `HttpDataProvider` вызывал
  глобальный `fetch` как собственный метод, и Chrome отвечал `Illegal invocation` ещё до сети;
  ошибка гасилась общим `catch` в комбобоксе (сохранение артикула, наоборот, показывало ошибку).
  Теперь `fetch` вызывается с глобальным объектом в качестве получателя, а подавленная ошибка
  поиска пишется в консоль.
- Дефект был не виден тестам: `fetch` в Node не проверяет получателя, а в jsdom-тестах провайдер
  подставляется явно. Регрессионный тест воспроизводит браузерное правило.

  `ArticleConstructiveField` → `ArticleErpField`, `ARTICLE_CONSTRUCTIVE_FIELDS` → `ARTICLE_ERP_FIELDS`,

### 2026-07-21 — групповое выделение и перенос стопок (`LKWkalk-dwc.6`)

- Рамка мыши по виду сверху: протяжка с пустого места пола выделяет все задетые стопки;
  Shift/Ctrl-клик добавляет или снимает одну, Escape снимает выделение.

  буфер раскладывает тайлы построчно и формы группы не хранит.

- Контракт движка `0.14.0` — добавлены `unplaceStacks`, `moveStacks`, `resolveGroupDrop`

### 2026-07-20 — каталог артикулов: единое поле-автокомплит (эпик `LKWkalk-8a0`)

Боль владельца: в строке позиции пресет и название дублировали друг друга (`rgv.8`), а правила
груза — вложение, ярусы, ориентация — вводились заново при каждой заявке, хотя для артикула они
неизменны. [Спека](superpowers/specs/2026-07-20-article-autocomplete-design.md),
[план](superpowers/plans/2026-07-20-article-autocomplete.md).

- **Каталог на сервере, не в `localStorage`.** Новая таблица `article` в SQLite (`apps/server`):
  конструктивные поля (`length`/`width`/`height`, оба прироста вложения), свободные правила
  (`rules_json`), провенанс. `DataProvider` получил `searchArticles(query)` /
  `upsertArticle(a)` — SPA по-прежнему не ходит в ERPNext напрямую.
- **Запирание — по полю, не по записи.** Изначальный план «заперто, если значение пришло из ERP»
  не выражал реальное правило: поле, однажды заполненное вручную, а затем никогда не тронутое
  ERPNext, не должно запираться. Добавлена колонка `erp_fields_json` — точный список полей,
  которые ERPNext когда-либо прислал для артикула; `Article.erpFields:
  readonly ArticleConstructiveField[]` в контракте. Заперты только перечисленные там поля;
  оба прироста вложения и `name` не запираются никогда — ERPNext их не поставляет. Спека
  исправлена этим же коммитом (была описана через `source`/`synced_at`, что не совпадает с кодом).
- **Комбобокс «Артикул»** (`apps/web/src/screens/SetupScreen.tsx`) заменяет `Select` пресетов и
  поле названия. Подсказки — из каталога сервера + встроенные EPAL с пометкой «стандарт»; выбор
  подставляет название, габариты, оба прироста и правила; клавиатура — `↑`/`↓`/`Enter`/`Esc`.
  Кнопка «Сохранить артикул в базу» / «Обновить артикул» пишет в каталог, ошибка сохранения
  показывается рядом с кнопкой, а не падает необработанным rejection.
- **Автопополнение при импорте заказа** — сервер делает upsert артикулов заявки с `source='erp'`,
  не трогая локально заданные правила однажды настроенного артикула.
- **Один прирост на режим вложения.** Переключение `pairwise`/`sequential` больше не подставляет в
  `stepHeight` ядра цифру от другого режима — каталог хранит оба прироста отдельно. Контракт
  движка не менялся.
- `apps/web/src/data/userPresets.ts` (каталог пресетов в `localStorage`, без артикулов, не
  участвовал в подсказках) удалён целиком.
- Закрывает `LKWkalk-rgv.8` (дубль поля названия) тем же изменением. Синк каталога с ERPNext по
  кнопке — вне рамок, заведена отдельная задача (зависит от `LKWkalk-k06`, кастом-поля габаритов
  в ERPNext ещё не созданы). Тестов: 436.

### 2026-07-17 — склад 1:1 и магнит постановки (эпик `LKWkalk-a5b`)

Батч правок владельца по живому UI. [Спека](superpowers/specs/2026-07-17-warehouse-buffer-design.md),
[план](superpowers/plans/2026-07-17-warehouse-buffer.md), [ADR 020](adr/020-magnet-drop-resolution.md).

- **Порядок секций** (`91u`): вид сбоку → вид сверху → склад. Кузов оказывается рядом с площадкой, между
  которыми ходят стопки. Попутно обезврежена мина: PNG подписывал разрезы из захардкоженного массива
  по позиции в DOM — перестановка молча поменяла бы подписи в файле, который уходит другому человеку.
  Подпись теперь берётся из `data-cutaway` самого svg.
- **Склад вместо полосы карточек** (`wxi`, `sqj`, `rue`): плитки — квадраты в масштабе **1:1** с кузовом,
  того же вида (общий примитив `StackShape`). На живом плане полоса была шестью одинаковыми карточками
  «EPAL 3 · 1000×1200 · ×18»; квадраты говорят это один раз и отвечают на вопрос, которого карточки не
  касались: влезет ли и как повёрнута. Кнопка `⟳` ушла — поворот переехал на жест кузова (выделить →
  ручка), плитки фокусируемы с клавиатуры, иначе поворот с клавиатуры пропал бы вместе с кнопкой.
  Погрузчик на фоне — **вид сверху**, 1:1, и только там, где не стоит на грузе.
- **Магнит + живая подсветка** (`crb`, `3kr`): `resolveDrop` в ядре (контракт **0.13.0**) ищет ближайшее
  допустимое место с **прижимом вплотную** — щель в 60 мм закрывается, а не сохраняется. Призрак
  показывает исход **до отпускания**: зелёный на месте посадки, красный на прицеле плюс красные контуры
  мешающих стопок. `placeStack`/`moveStack` **остались строгими** — магнит вызывается отдельно, иначе
  API/MCP потерял бы примитив «поставь ровно сюда».
- **D1 + D3** (`5i4`): `Metrics` ушёл с листа, «Заполнение объёма» стало пятой фигурой мета-полосы.
  Полоса — единственный источник цифр; «не размещено» звучит дважды вместо трёх раз; PNG получил объём
  даром. Мёртвые ключи `ladeplan.fig.load` и `results.unplaced` сняты из словарей.
- **Найдено только браузером:** равенства `viewBox` для масштаба 1:1 **недостаточно** — свои `px-4` и
  рамка у секции склада давали 0.09632 против 0.09882 px/мм при совпадающем `viewBox`. Убраны;
  прокрутка склада отвергнута (полоса прокрутки сужает svg и уводит масштаб). Подробности — поправка
  в §4.2 спеки.
- Проверено в Chrome: масштаб идентичен; круг «вынуть → склад → вернуть» даёт 140 → 156 → 140 единиц
  (баланс сохранён), 0 нарушений геометрии на каждом шаге. Тестов: 370 → 400.

### 2026-07-17 — вид сбоку: глубина ряда по перекрытию проекций (`LKWkalk-1t2`)

Наблюдение владельца: «вид сбоку иногда путает передний и задний ряд». Воспроизведено тестом, диагноз
подтверждён — [спека](superpowers/specs/2026-07-17-side-view-depth-design.md).

- **Корень:** `sideRects` ранжировала глубину только среди стопок с **совпадающим `x`**, но на виде
  сбоку перекрываются стопки с пересекающимися **интервалами** по длине. Две стопки в разных рядах,
  стоящие внахлёст, получали обе `depth = 0`: дальняя не притенялась, а при равной глубине порядок
  отрисовки оставался порядком `placements` — дальняя могла закрасить ближнюю. Отсюда и «иногда»:
  на однородном грузе ряды выровнены по `x`, и старое правило работало.
- **Лечение:** глубина = число стопок, которые реально загораживают эту (перекрытие по интервалу +
  бо́льший `y`). Порядок отрисовки — по возрастанию `rowY`, тай-брейк по `x`. Сортировка по глубине
  снята: «загорожена двумя» не означает «дальше, чем загороженная одной» — это счётчик, а не порядок.
- **Попутно:** одиночная стопка в дальнем ряду больше не притеняется без причины (`depth = 0`).
- **D2:** дальний ряд гасится только заливкой и штриховкой, контур остаётся в полную силу. При
  сплошном `opacity 0.4` низкие дальние стопки исчезали — четвертьпаллета это 864 мм из 2650. На
  проде область ~6 200…7 800 мм по длине была пустой белой; теперь ряд виден.
- Контракт не менялся (правка целиком в UI-проекции). Тестов: 365 → 370.

### 2026-07-17 — разбор плотности режима «Сбоку» (`u02`): не дефект, а потолок shelf-модели

Вопрос владельца по живому плану: «то, как сейчас рассчитывается план для загрузки сбоку — всё верно?»
Замер: `rear` 704 поддона / 92.9 % пола, `side` 600 / 71.1 % (`strict` — 542 / 56.6 %), нарушений
геометрии 0 везде. **Раскладка корректна**, правила груза целы — это потолок эвристики, а не ошибка.

- **Корень:** в `side` ось роста полок — ширина (2430) → помещается ровно две полки, их глубину
  задаёт самый толстый тип (2×1200 = 2400 из 2430), и всё мелкое дарит разницу в карман. В `rear`
  полки растут вдоль 13,6 м — их до одиннадцати, каждая по своему содержимому. Карманы **этим
  грузом неустранимы**: единственный остаток — EPAL 3 (`rotation: 'none'`, нужна глубина 1200),
  карманы — 600.
- **Замерено и отвергнуто:** политика «мелкая полка» (не выигрывает ни разу; форсированная — хуже:
  542 против 600); поворот остатка в кармане (0 прироста — остаток не вращается); полки по классам
  глубины (−198 поддонов на бумаге). Код после экспериментов **откачен**, движок не менялся.
- **[ADR 012](adr/012-loading-modes.md) дополнен поправкой:** утверждение «ёмкость симметрична»
  верно только для однородного груза (эталоны EUR 34 / GB 20 держатся), для смешанного — нет.
  Проверен и тай-брейк `combined` (сравнивает число стопок, а не поддонов): контрпример не найден ни
  случайным поиском (3000 нагрузок), ни целевым.
- **Настоящее лечение** — 2D max-rects вместо полок: заведён `LKWkalk-i1p`. Ценность ограничена:
  дефолт «Сзади и сбоку» уже отдаёт плотнейший проход (704), потеря достаётся только Tautliner.

### 2026-07-17 — буфер стопок: рабочая область для ручной правки (эпик `dwc`, задачи `dwc.3`, `dwc.4`)

Боль владельца: *«если места не хватает — программа не позволяет повернуть стопку, а затем если есть
ещё не загруженные, которые теперь влазят, их неоткуда взять»*.

- **Полоса-буфер под видом сверху** (решение владельца; боковая панель отвергнута — кузов 13,6 м,
  панель съела бы ширину там, где идёт работа). Плитки = **стопки**, а не штуки: цвет и штриховка
  заказа, форма футпринта, габариты, `×N`. Экран-онли: печать и PNG-экспорт её не видят
  (экспорт берёт только `svg[data-cutaway]`).
- **Двусторонний перенос.** Стопку из кузова тащим на полосу → `unplaceStack` (место освободилось →
  можно повернуть). Плитку из полосы тащим в кузов → `placeStack`. Точка сброса — середина стопки,
  угол прижимается к габариту кузова: целиться в угол значит «поставь в угол», а не «пусть свисает».
- **Поворот в буфере** — по типу груза (стопки одного типа взаимозаменяемы, а их порядок в списке
  меняется после каждой правки; ориентация на индексе списка досталась бы случайной стопке).
  Для `rotation: 'none'` кнопка выключена.

### Documentation
- **[ADR 018](adr/018-fork-access-orientation.md) — доступ погрузчика как ограничение ориентации
  (эпик `LKWkalk-4bj`, `4bj.10`).** Новое требование владельца: 2-way поддоны (доступны только с двух
  противоположных сторон) нельзя разворачивать недоступной стороной к двери — жёсткий физический
  инвариант. `CargoType.forkAccess: 'all4' | 'twoSides'` + `forkAxis: 'length' | 'width'` (контракт
  **0.11.0**, аддитивно, default `all4` = текущее поведение). При `rear`/`side` 2-way поддон пиннится
  в одну ориентацию, при `combined` — свободен. В UI — один список «Ausrichtung» (Fixiert / Drehbar
  4 Seiten / Drehbar 2 Seiten); `rotation: 'full'` убирается из UI (в упаковщике уже ≈ yaw, ADR 013).
  Реализуется до плотной эвристики (`4bj.8`), т.к. меняет `chooseOrientation`.
- **Умная раскладка — спека и решения (эпик `LKWkalk-4bj`, `4bj.6`).** Разбор упаковщика показал, что
  «полки режутся под первый тип» — лишь **один из трёх** независимых источников карманов: храповик
  `shelfDepth` (глубина полки растёт и не убывает, backfill отсутствует), next-fit (открытие новой
  полки навсегда закрывает предыдущую) и выбор ориентации против пустого региона; четвёртый источник
  — срезы зон заказов — ортогонален. Половина задачи оказалась уже реализованной в движке и не
  выведенной в UI (`Load.loadingMode`, ADR 012). Задача разведена на `4bj.7` (вывести режимы в UI,
  дёшево), `4bj.8` (плотная эвристика, движок) и `4bj.9` (тумблер смежности зон).
  Спека — `docs/superpowers/specs/2026-07-16-smart-layout-design.md`.
- **[ADR 017](adr/017-dense-floor-heuristic.md)** — плотная напольная эвристика: best-fit вместо
  next-fit, backfill карманов, ориентация против остаточного места. Устранено расхождение ADR ↔ код:
  предписанный ADR 004 сорт футпринтов по убыванию площади **никогда не был реализован** и признан
  отменённым — порядок входа несёт семантику приоритета заявки (`floor.test.ts:141`), а плотность
  достигается без переупорядочивания запросов. Зафиксировано, что интерфейс `Packer` существовал
  только в документации; фактический шов — `packFloor` (правки `design.md`, ADR 004, `CLAUDE.md`).
- **[ADR 016](adr/016-order-grouping-density-toggle.md)** — `Load.orderGrouping: 'strict' |
  'densityFirst'` (контракт **0.10.0**, аддитивно, default `strict`): явное ослабление смежности зон
  заказов ради плотности. ADR 011 остаётся дефолтом; «заказ выгружается целиком» становится свойством

### Fixed
- **Вид сбоку показывал задний ряд передним (`LKWkalk-4bj.2`).** Глубина ранжировалась по возрастанию
  `y`. Конвенция зафиксирована: вид сбоку снят от нижнего края вида сверху (наблюдатель на `y = width`),
  значит **больший `y` — ближний ряд** (`depth 0`, полная непрозрачность), меньший — дальше (тусклее).
  Добавлено поле `CutRect.rowY`, тест сторожит конвенцию.
- **Click-outside съедал клик по кнопке (`LKWkalk-4bj.1`).** Панель «Verschachtelt» закрывалась на
  `mousedown` → DOM сдвигался до `mouseup`, и браузер отправлял `click` на предка, а не на нажатую
  кнопку («+ Position»/«+ Auftrag») — её обработчик не срабатывал. Слушаем `click`: обработчик кнопки
  (делегирование React на root) отрабатывает первым, затем панель закрывается.
- **Вёрстка строки позиции ломалась на длинных RU-лейблах (`LKWkalk-e5i`).** Ротация «Только вокруг
  вертикальной оси» распирала `nowrap`-строку → горизонтальный overflow страницы (на узком окне
  переполнял и DE, поэтому возврат на DE не помогал). Поле ротации — фиксированная ширина +
  `truncate` (`Select` базово `min-w-0/truncate`), строка переносится ниже 1280 px (`xl:flex-nowrap`);
  плюс запас по ширине (поджатые гэпы, меньшая min-ширина имени), чтобы контролы не упирались в край.
- Колонка «парами» (pairwise): per-tier `z` теперь достигает истинной высоты штабеля
  `H + k·(H + h_д)` вместо сжатого `t·h_д` — корректный вид сбоку и `volumeFillPercent`
  (`LKWkalk-qrd.22`). Геометрия остаётся валидной (column-aware, ADR 014).
